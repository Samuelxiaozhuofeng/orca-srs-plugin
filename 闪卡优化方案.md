## FC-01 每日新卡和每日复习上限生效

**优先级：P0**

### 问题说明

设置中已经提供 `review.newCardsPerDay` 和 `review.reviewCardsPerDay`，但 `buildReviewQueue()` 没有读取或应用它们。用户设置每日上限后，当前队列仍可能加入全部到期卡和全部新卡。

### 技术依据

- `src/srs/settings/reviewSettingsSchema.ts:40`：定义每日新卡上限。
- `src/srs/settings/reviewSettingsSchema.ts:46`：定义每日复习卡上限。
- `src/srs/cardCollector.ts:398`：`buildReviewQueue()` 只筛选并按 2:1 交织，没有限制数量。
- `src/srs/cardCollector.ts:451`：子卡片展开可能在基础队列之后继续增加数量。
- `src/components/SrsReviewSessionDemo.tsx:454`：动态检查也会重新构建队列并追加卡片。

### 实施要求

- 为队列构建提供明确的限制输入，避免核心函数隐式依赖全局插件状态。
- 明确旧卡上限和新卡上限是全局限制还是牌组内限制，并在测试中固定当前选择。
- 明确限制发生在子卡展开前还是展开后。
- 推荐把“正式到期根卡片限额”和“随根卡展开的辅助子卡”分开计数，避免子卡随机挤掉用户应复习的根卡；最终规则必须写入模块文档。
- 定时检查、手动检查和 Again 后短期重新入队均不得绕过上限。
- 对负数、`NaN`、小数和异常大的设置值进行校验，错误应可见并回退到明确默认值。
- 不影响重复复习/专项训练模式；该模式是否受限应明确写入行为说明。

### 验收标准

- [ ] 每日新卡数量不超过设置值。
- [ ] 每日旧复习卡数量不超过设置值。
- [ ] 指定牌组复习时限制在该牌组范围内计算。
- [ ] 动态追加不会突破已确定的会话限额。
- [ ] 子卡展开与限额关系有确定规则和测试。
- [ ] 无效设置值不会导致队列异常或静默失效。
- [ ] 更新 `模块文档/SRS_复习队列管理.md`。

### 建议测试

- 新卡 50 张、上限 30，队列只能包含 30 张新卡。
- 旧卡 300 张、上限 200，队列只能包含 200 张旧卡。
- 同时存在新旧卡时仍保持确定的交织规则。
- 动态检查发现新卡时，剩余额度为 0 不得追加。
- 子卡展开后的数量符合选定规则。
- 建议新增 `src/srs/cardCollector.test.ts` 或独立的队列策略测试文件。

## FC-02 指定牌组会话保持牌组隔离

**优先级：P0**

### 问题说明

初始复习队列会应用 `deckFilter`，但复习会话运行期间的定时检查和手动检查会从全库收集卡片，没有再次应用牌组过滤。牌组会话运行超过一分钟后，其他牌组的到期卡可能被追加进来。

### 技术依据

- `src/components/SrsReviewSessionRenderer.tsx:94-100`：初始队列正确应用 `deckFilter`。
- `src/components/SrsReviewSessionDemo.tsx:458-480`：定时检查直接对全库卡片调用 `buildReviewQueue()`。
- `src/components/SrsReviewSessionDemo.tsx:798-820`：手动检查同样没有牌组过滤。

### 实施要求

- 把本次会话的牌组范围作为显式参数传给复习会话组件。
- 初始加载、定时刷新、手动刷新和短期卡重新入队必须使用同一范围判断。
- 不要在子组件中重新读取可能已经变化的全局临时过滤值，避免会话中途范围漂移。
- “今日全部复习”使用无牌组限制；单牌组复习使用固定牌组限制。
- 专项训练和困难卡复习应继续使用它们自己的固定卡片集合。

### 验收标准

- [ ] 单牌组会话运行期间不会加入其他牌组卡片。
- [ ] 手动点击“检查新到期卡片”不会破坏牌组范围。
- [ ] 全部牌组复习仍可追加任意牌组的到期卡。
- [ ] 会话范围通过 Props 或明确会话对象传递。
- [ ] 有组件或纯逻辑回归测试。

### 建议测试

- 会话启动后修改全局 deck filter，不改变已启动会话范围。

## FC-03 复习日志可靠落盘

**优先级：P0**

### 问题说明

复习日志使用一秒延迟批量写入。连续评分会不断重置定时器；插件卸载时没有主动刷新。`flushPendingLogs()` 还会在持久化成功前清空 `pendingLogs`，一旦写入失败，这批日志无法自动重试。

评分更新 SRS 状态成功但日志失败时，当前调用还会返回成功，用户和统计模块无法知道日志缺失。

### 技术依据

- `src/srs/reviewLogStorage.ts:23-27`：全局 pending 缓冲区和延迟时间。
- `src/srs/reviewLogStorage.ts:108-129`：写入前清空 pending。
- `src/srs/reviewLogStorage.ts:135-142`：定时器回调没有显式错误处理。
- `src/srs/reviewCardGrading.ts:157`：`void saveReviewLog(...)`，调用方不等待结果。
- `src/main.ts:116-126`：插件卸载时没有刷新日志。

### 实施要求

- 持久化成功后才能从 pending 缓冲区移除相应记录。
- 失败时保留待写日志，并输出清晰错误；需要定义有限重试或下一次 flush 重试策略。
- 避免定时器 Promise 产生未处理拒绝。
- 插件 `unload()` 中调用 `flushReviewLogs(pluginName)`，并明确卸载清理的执行顺序。
- 复习会话正常关闭/完成时也应触发一次 flush，缩短风险窗口。
- 明确 SRS 状态写入成功但日志写入失败的返回语义。至少向用户显示“评分已保存，但统计日志保存失败”，不能宣称完全成功。
- 如保留批量写入，必须防止重复日志；日志 ID 应支持幂等合并或去重。

### 验收标准

- [ ] 连续快速评分不会丢日志。
- [ ] 存储首次失败后，pending 日志仍存在并可再次写入。
- [ ] 定时 flush 失败不会产生未处理 Promise rejection。
- [ ] 插件卸载前会尝试完成日志落盘。
- [ ] 部分成功状态对用户可见。
- [ ] 重试不会写出重复日志。

### 建议测试

- 使用 fake timers 连续保存多条日志，最终一次 flush 全部写入。
- mock `setData` 首次失败、第二次成功，验证记录未丢且未重复。
- 验证 `unload()` 调用 flush。
- 验证评分成功但日志失败时返回 warning 或明确错误状态。

## FC-04 列表卡日志不被删除清理误删

**优先级：P0**

### 问题说明

列表卡评分时使用列表子条目 ID 作为日志 `cardId`。启动清理只把收集到的 SRS 父卡块 ID 视为有效，因此列表条目日志可能被误判为已删除卡片日志并清除。

### 技术依据

- `src/srs/reviewCardGrading.ts:142`：List 卡日志 ID 使用 `listItemId`。
- `src/srs/deletedCardCleanup.ts:31-48`：有效 ID 只来自 `collectSrsBlocks()`。
- `src/main.ts:102-109`：插件启动三秒后自动执行清理。

### 实施要求

- 不要只用“是否带 #card 标签”判断日志目标是否有效。
- 配合 FC-05 设计统一日志身份，至少保留父卡 ID和变体/条目 ID。
- 对列表卡同时验证父卡和子条目：父卡存在且子条目仍属于有效列表时日志有效。
- 对 Basic、Cloze、Direction、Choice 和 List 采用明确且可测试的存在性规则。
- 旧版只有 `cardId` 的日志必须兼容，无法确定时优先保留而不是误删。
- 清理过程遇到读取失败时不得把“无法确认”当成“已删除”。

### 验收标准

- [ ] 有效列表条目日志不会被清理。
- [ ] 已删除的列表条目日志可以被正确识别。
- [ ] 后端读取暂时失败不会造成日志删除。
- [ ] 旧日志仍能读取和保留。
- [ ] 清理数量和错误信息真实可见。

### 建议测试

- 父列表卡和子条目都存在，清理数量为 0。
- 子条目删除，日志按定义被清理。
- 父卡删除，相关列表日志被清理。
- `get-block` 抛错时日志不被删除，并返回/记录失败信息。

## FC-05 建立稳定的卡片变体身份

**优先级：P0**

### 问题说明

当前复习日志只记录一个数字 `cardId`。同一父块里的 Cloze c1/c2、Direction forward/backward 无法区分；List 卡则改用子条目 ID。困难卡分析通过字符串包含判断匹配日志，会产生串卡或匹配不到的问题。

### 技术依据

- `src/srs/types.ts:170` 附近：`ReviewLogEntry` 只有 `cardId`。
- `src/srs/reviewCardGrading.ts:142-154`：不同卡型写入日志的身份规则不一致。
- `src/srs/difficultCardsManager.ts:80-110`：卡片键和日志键通过字符串包含匹配。
- `src/srs/childCardCollector.ts`：已有复习卡 key 概念，可作为设计参考，但不要直接假设可复用。

### 实施要求

- 为日志增加结构化身份字段，建议至少包含：父卡 `blockId`、卡片类型、`clozeNumber`、`directionType`、`listItemId`。

- `cardId` 的兼容语义必须保留，避免破坏现有统计和旧数据。

- 困难卡分析必须使用精确匹配，不再使用 `includes()`。
- 删除清理、事件广播、困难卡、统计筛选使用同一身份定义。

## FC-06 防止返回上一张后重复评分

**优先级：P0**

### 问题说明

当前“回到上一张”只是把索引移回历史位置。已评分卡可以再次显示答案并评分，从而第二次更新 FSRS 状态、增加 reps/lapses 并写入重复日志。

### 技术依据

- `src/components/SrsReviewSessionDemo.tsx:508-588`：评分会立即持久化。
- `src/components/SrsReviewSessionDemo.tsx:845-857`：返回上一张只修改索引，没有只读保护或撤销逻辑。
- 各卡型 ReviewRenderer 都暴露“回到上一张”按钮。

### 实施要求

- 第一阶段推荐实现“只读回看”：已评分卡返回后可查看题目/答案和上次评分，但评分、推迟、暂停操作禁用。
- 为会话历史记录已评分状态、评分结果和必要的展示信息，不能仅存索引。
- 回到未评分但跳过的卡片时，是否允许评分需要明确规则。
- 如果选择实现真正撤销，必须原子恢复此前 SRS 全状态、列表推进状态和日志，并处理撤销失败；该方案复杂度更高，不能只删除日志。
- 快捷键也必须受只读状态限制，不能仅禁用按钮。

### 验收标准

- [ ] 返回已评分卡后不能再次触发评分。
- [ ] 数字键和空格键不会绕过只读限制。
- [ ] 原卡片评分结果和到期状态不会被二次修改。
- [ ] 回看后继续前进不会破坏历史索引。

## FC-07 保留卡片重置次数

**优先级：P0**

### 问题说明

重置操作会增加 `resets`，但下一次评分生成的新状态没有继承旧值。存储层对缺失值使用 `0`，导致重置次数在评分后归零。

### 技术依据

- `src/srs/algorithm.ts:141-151`：重置状态增加次数。
- `src/srs/algorithm.ts:154-175`：`nextReviewState()` 返回值未包含 `resets`。
- `src/srs/storage.ts:160`、`:355`：缺失时保存为 0。
- `src/components/SrsFlashcardHome.tsx:389` 附近：主页显示重置次数。

### 实施要求

- `nextReviewState()` 必须继承 `prevState?.resets ?? 0`。
- Basic、Cloze 和 Direction 走同一算法路径，分别增加回归测试。
- List 条目使用普通卡状态，也要验证。
- 预览评分调用 `nextReviewState()` 时不得改变原状态。

### 验收标准

- [ ] 重置一次后评分，`resets` 仍为 1。
- [ ] 重置多次后评分，累计值保持。
- [ ] 未重置卡片仍为 0。
- [ ] 所有支持卡型行为一致。

### 建议测试

- 为 `algorithm.ts` 增加纯函数测试。
- 分别验证 `resetCardSrsState`、`resetClozeSrsState`、`resetDirectionSrsState` 后评分。

## FC-08 接通选择题答题统计

**优先级：P1**

### 问题说明

项目已有 `choiceStatisticsStorage.ts` 和选项错误频率显示组件，但复习入口没有向 `ChoiceCardReviewRenderer` 传递 `onAnswer`，实际答题不会写入统计。

### 技术依据

- `src/components/SrsCardDemo.tsx:697-713`：选择题渲染器未传 `onAnswer`。
- `src/components/ChoiceCardReviewRenderer.tsx:149-188`：提交时会调用可选的 `onAnswer`。
- `src/srs/choiceStatisticsStorage.ts:82-103`：已有保存函数。
- `src/components/ChoiceStatisticsIndicator.tsx`：依赖已保存记录显示频率。

### 实施要求

- 在选择题实际提交答案时构造 `ChoiceStatisticsEntry` 并持久化。
- 记录选中 ID、正确 ID、是否完全正确和时间戳。
- 保存失败必须可见；明确是否允许用户继续评分，并以 warning 告知统计未保存。

- `src/hooks/useSessionProgressTracker.ts:110-123`：挂载时自动恢复。

## FC-10 统一复习时长口径
