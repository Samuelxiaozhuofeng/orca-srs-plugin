## 4. 所有 Agent 必须遵守的执行规则

1. 开始前先读根目录 `AGENTS.md`、`plugin-docs/modules.md`、对应 Orca API 文档和本工作包列出的模块文档。
2. 一次只处理一个工作包，不顺手重构其他模块，不覆盖无关用户改动。
3. 涉及块、属性、标签、缓存、索引、编辑器命令或数据库一致性时，修改前先提供只读 Console 脚本，并取得真实 Orca 输出。
4. Console 证据必须同时包含 `orca.state.repo`、ISO 时间、目标 ID、`orca.state.blocks` 快照和经本地文档确认的 backend 独立读取。
5. 未经同意不得提供或运行修改数据的 Console 脚本；命令返回成功不等于写入确认，必须 backend 重读。
6. 所有块属性写入后，后续读取依赖新值前必须调用匹配的缓存失效函数；IR 使用 `invalidateIrBlockCache`。
7. 卡片身份只能通过 `src/srs/cardIdentity.ts` 生成和比较，不得手拼字符串或使用子串匹配。
8. 后端错误必须显示为错误或可重试状态，不得伪装为 `null`、空数组、无卡片或成功完成。
9. 行为变更后同步对应 `模块文档/`、`模块文档/README.md`；真实数据确认的回归应写入 `模块文档/问题经验.md`。
10. 每个工作包先跑聚焦测试，再跑 `npx tsc --noEmit` 和全量测试。`npm run build` 仅在仓库内产出 `dist/`（无外拷）；不得手工编辑 `dist/`。正式 `release:zip` 仍受 readiness Go、干净工作树与 `v1.0.0` tag 门禁约束。
11. `dist/` 只能由构建生成，不得手工编辑。
12. 最终报告必须分别写“自动化验证”和“Orca 真机验证”，两者不能互相替代。

## 7. 计划维护规则

- Agent 开始工作时只把自己认领的工作包标为“进行中”。
- 自动化测试通过但缺少真机证据时，工作包状态只能是“待真机验证”。
- 工作包完成后在标题下记录完成日期、提交 hash 和验收人，不删除原始验收标准。
- 若实现改变了工作包边界，先更新本计划并说明原因，再修改代码。
- 新发现的 P0/P1 问题先加入本计划并指定工作包，不在当前 Agent 范围中静默顺手修复。
