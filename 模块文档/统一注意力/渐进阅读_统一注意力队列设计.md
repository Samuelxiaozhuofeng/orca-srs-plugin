> **文档类型：产品目标 + 技术设计 + 分阶段开发计划（尚未落地）**

>

## 0. 一句话目标

用户每天只需要：

1. 打开“今日学习”；
2. 阅读系统送来的章节、Topic、Extract 和到期记忆卡；
3. 对当前内容执行“下一篇、摘录、挖空、重要性、完成”等局部动作；
4. 时间到后离开。

用户不需要：

- 记住每本书读到哪里；
- 手工安排每个章节或摘录的回归日期；
- 担心同一天产生的大量摘录未来集体到期；
- 每天整理 backlog 才能开始阅读；
- 在多个面板之间判断“今天到底应该做什么”；
- 为了避免遗忘而把所有内容立即制作成卡片。

系统必须提供的核心承诺是：

> **重要内容会较快回来；低重要性内容可以晚，但不会无声消失；每日工作量有限；所有未完成内容的去向可解释、可恢复。**

## 1. 产品问题：用户真正需要的不是“随机算法”

### 1.1 表层需求

表层描述通常是：

- 导入十本书，让算法分散章节；
- 今天做二十个摘录，不要未来某天二十个一起回来；
- Topic 太长时读一部分就切换；
- Extract 回来后继续精炼，必要时制作 Cloze；
- 每天打开队列一直处理即可。

### 1.2 深层需求

这些需求共同指向一个更完整的产品：

**可信的注意力调度系统**

“可信”至少包含六层含义：

| 信任 | 用户预期 | 系统责任 |
| --- | --- | --- |
| 收纳可信 | 可以放心导入很多内容 | 导入规模不直接等于明日负担 |
| 回归可信 | 放走后以后仍会回来 | 每个 active 条目都有可解释的再出现路径 |
| 容量可信 | 今天的任务不会无限增长 | 今日预算跨面板、跨重开一致 |
| 现场可信 | 随时可停 | 保存阅读断点、当前会话和动作结果 |
| 漏斗可信 | 不必立刻决定“记不记” | Topic → Extract → Item 可渐进加工 |
| 错误可信 | 失败不会被伪装成完成 | 读取、排期、日志和写入失败保持可见、可重试 |

### 1.3 为什么“加一点随机”不够

纯随机可以降低机械扎堆，但不能解决：

- 多本书各自分散后在同一日期相互重叠；
- 连续多日各自产生一批 Extract，多个批次的波峰叠加；
- 某天 SRS 已经很重，IR 仍继续把大量内容排到当天；
- 低重要性内容长期被高重要性内容压住；
- 同一来源连续出现，造成认知疲劳；
- 用户每天实际只有 20 分钟，但排期只按“张数”计算。

目标应是：

> **稳定随机 + 容量感知 + 重要性感知 + 等待补偿 + 来源多样性 + 明确约束。**

## 2. 与 SuperMemo 的关系

### 2.1 借鉴的体验与架构

本文借鉴以下公开、可观察的思想：

- Topic 与 Item 都会进入日常学习流；
- Topic/Extract 允许多次回来后再逐步加工；
- interval/due 决定何时重新成为候选；
- priority 决定有限容量下先处理什么；
- 同源 Extract 需要分散；
- backlog 需要自动治理；
- 用户主要处理当前条目，而不是手工维护完整日历。

### 2.2 不声称复刻的部分

本文不声称复刻：

- SuperMemo 17/18 的闭源记忆算法；
- Topic A-Factor 的私有更新权重；
- Extract 初始间隔的精确公式；
- priority 到 interval 的私有映射；
- Auto-postpone 的全部内部参数；
- Outstanding 中 Topic/Item 混排的精确随机机制。

### 2.3 本项目的职责边界

建议继续保持：

- **普通主动回忆卡**：由现有 FSRS 路径调度；
- **Topic / Extract**：由 IR 阅读调度管理；
- **统一注意力队列**：只负责在有限时间内选择和交织两类条目；
- **Extract 转 Item**：是 IR 与 FSRS 的分界动作。

不得为了界面统一，把 Topic 的“读过”伪装成 FSRS 的 Remember 评分。

### 3.2 当前排期模型

#### 普通 Topic / Extract

| 类型 | priority 低 → 高的基础间隔 | “下一篇”后的增长 | 上限 |
| --- | --- | --- | --- |
| Topic | 约 14 天 → 2 天 | `×1.25` | 60 天 |
| Extract | 约 7 天 → 1 天 | `×1.35` | 30 天 |

#### 顺序 Book IR 当前章

- 基线：`1 + 2 * (1 - priority / 100)`；
- priority 0 / 50 / 100 约为 3 / 2 / 1 天；
- 连续无断点进展时逐步拉长；
- 上限约 6 天。

#### 新 Extract 首次错峰

- 新卡 interval 有稳定的只向后抖动；
- 同一来源 Topic、同一自然日、更早创建的兄弟 Extract 产生 `queueDelayDays`；
- queue delay 只加入首次 `due`，不写入长期 `intervalDays`。

### 3.3 当前队列模型

当前时间盒会话：

1. 收集本地日 `due <= 今天` 的 Topic / Extract；
2. 依据时间预算和 `dailyLimit` 选择；
3. 保护高重要性旧积压；
4. 满足 Topic floor；
5. 限制新 Extract 比例；
6. 使用小比例稳定探索；
7. 可选混入到期、非新、未暂停的 SRS 卡；
8. 形成内存快照。

### 3.4 当前已知边界

| 边界 | 当前实际含义 |
| --- | --- |
| `dailyLimit` | 单次队列硬上限，不是跨会话每日剩余额度 |
| IR session | React 内存快照，不是持久化 frozen descriptor |
| mixed SRS | 会话层交织，尚未完整复用普通 SRS 的 frozen scope / daily budget |
| 分散章节 | 单本书内部均匀 + jitter，不读取全局未来负载 |
| 分散 Extract | 同源同日 batch 错峰，不读取跨日、跨来源未来负载 |
| overdue | 当前选择主要区分逾期/今天，没有完整的等待年龄补偿 |
| 自动治理 | 主会话加载只读；资料库按钮触发溢出推后 |
| 自适应 cadence | 普通 Topic/Extract 仍用固定增长倍率；动态 A-Factor 未落地 |

## 4. 差距定义

### 4.1 局部分散不等于全局均衡

一本书十章分散在 30 天内，只保证这一本书内部相对铺开。若同时导入十本书，每本书都可能把章节排到相似日期。

同理，同一天同一 Topic 的 Extract 会错峰，但连续三天每天产生 20 个 Extract，会得到三条相近的错峰曲线；这些曲线可能在未来重叠。

### 4.2 单次限制不等于今日承诺

如果 `dailyLimit=30`，处理完后重新打开又能获得一批 30 张，用户无法把“今天有多少任务”理解为稳定承诺。

### 4.3 有 due 不等于保证被选择

一个低重要性条目可以长期保持逾期，但如果每天都被更高价值内容占满，它可能持续未入选。系统需要区别：

- 用户主动推后；
- 尚未到期；
- 已到期但本次容量不足；
- 连续多次有资格却未被选择；
- 读取失败导致没有参与选择。

### 4.4 混合展示不等于统一预算

把 IR 与 SRS 交织到一个数组里，只解决了视觉连续性；它没有自动解决：

- SRS 今日 new/review remaining；
- frozen review scope；
- 同一 cardKey 是否已在另一会话消费；
- IR 当日剩余额度；
- 刷新后的去重与恢复。

### 4.5 自动治理与只读启动存在合理张力

会话启动不应偷偷改 due；但如果永远要求用户手动处理溢出，系统又无法达到低决策体验。

正确方向不是恢复“每次打开都自动改排期”，而是设计：

- 明确的每日治理时机；
- 冻结队列后再计算旧积压；
- 仅处理真正的旧积压；
- 一日一次；
- 可解释、可撤销；
- 实际成功/失败可见。

## 5. 目标体验

### 5.1 单一日常入口

主入口建议命名为：

**今日学习**

进入后直接显示：

- 今天剩余的预计时间；
- 已完成的阅读/复习数量；
- 当前条目；
- 必要时的非阻断警告。

默认不要求用户先进入资料库、查看 backlog 或调整参数。

### 5.2 连续注意力流

一个目标会话可能是：

```text
《书 A》第 3 章 Topic
→ 三天前的 Extract
→ 到期 Cloze
→ 《书 B》第 8 章 Topic
→ 昨天的 Extract
→ Direction 卡
→ 网页 Topic
```

顺序由系统决定，但需满足：

- focus 优先；
- 到期与预算合法；
- 高重要性受保护；
- Topic 不被 Extract 永久淹没；
- 新 Extract 不挤满会话；
- 同来源连续出现受到限制；
- 低重要性条目获得有界等待补偿；
- SRS 不突破其权威预算。

### 5.3 用户只做局部决策

| 动作 | 用户意图 | 系统责任 |
| --- | --- | --- |
| 下一篇 | 我已经推进到这里，之后再回来 | 保存断点，更新 cadence，安排 due |
| 摘录 | 这段值得继续加工 | 创建 Extract，继承来源并全局错峰 |
| 挖空 | 这部分已经适合主动回忆 | 创建/更新 Cloze，由 FSRS 接管 Item |
| 重要性 | 长期投入应增加/减少 | 改价值信号，不伪造阅读动作 |
| 推后 | 现在不适合处理 | 只移动 due，不增长 intentional interval |
| 完成 | 不再需要作为 IR 条目回来 | 清 IR 调度，保留正文与已有卡片 |
| 关闭 | 今天先到这里 | flush 断点与日志，冻结状态可恢复 |

### 5.4 时间到的语义

时间盒到期时：

- 不打断当前阅读；
- 通知用户已达到目标时间；
- 可继续，也可立即关闭；
- 未处理条目不自动视为失败；
- 是否进行旧积压治理由独立策略决定，而不是把“未选中”伪装成已推后。

## 6. 核心设计原则与不变量

### 6.1 Cadence 与 Selection 分层

- **Cadence**：这张内容什么时候重新具备资格；
- **Selection**：今天有限时间里先处理哪些有资格内容。

Selection 不得偷偷修改 cadence。未被选中不等于已成功 postpone。

### 6.2 重要性与临时时机分离

- `ir.priority` 表示长期注意力价值；
- `ir.due` 表示下一次允许出现的时机；
- `ir.intervalDays` 表示真实处理形成的 intentional cadence；
- 手动或自动 postpone 只移动 due，不污染 interval。

### 6.3 容量使用预计时间，不只使用张数

Topic、Extract 和 SRS 的处理成本差异很大。预算以秒/分钟为主，张数作为安全硬上限。

### 6.4 高价值保护，但不允许永久饥饿

priority 必须主导选择；等待补偿必须有上限。低 priority 不应因无限逾期最终压倒所有高 priority，但需要有限的 mercy / exploration 路径。

### 6.5 随机必须稳定且受约束

- 同一 frozen session 重放结果一致；
- 本地自然日切换后允许变化；
- 随机只在相近价值候选间轻微扰动；
- 不允许随机越过保护区和硬预算。

### 6.6 冻结描述符优先于动态方便

同一会话的 scope、预算、比例和 seed 必须冻结。未来若支持动态补队列，也必须遵守原 descriptor 与剩余额度。

### 6.7 数据错误不能伪装为空队列

- 查询失败 ≠ 没有内容；
- enqueue 成功 ≠ 日志已落盘；
- 命令返回成功 ≠ backend 已确认写入；
- 计划数量 ≠ 实际成功数量。

### 6.8 保持现有工程不变量

后续实现必须继续遵守仓库合同：

- 普通卡 FSRS 设置走 validated path，weights 恰为 21 个数；
- block property 写后使对应 cache 失效；
- card identity 只通过 `src/srs/cardIdentity.ts`；
- review session 继续使用 frozen descriptor；
- 批量块读取、child expansion 有界；
- 错误保持可见和可重试；
- Orca UI runtime React 来自 `window.React`。

## 7. 目标分层架构

```text
Ingestion Layer
  EPUB / Web / 手动 Topic / Extract 创建
        ↓
Cadence Layer
  初始窗口、intentional interval、A-Factor、due
        ↓
Eligibility Layer
  active + due <= policy boundary
        ↓
Daily Budget Layer
  repo + local day + unique consumption ledger
        ↓
Selection Layer
  priority + overdue aging + starvation + diversity + constraints
        ↓
Frozen Session Layer
  descriptor + entries + progress + remaining budgets
        ↓
Presentation Layer
  连续阅读 / SRS 复习 / 当前动作
        ↓
Overload Layer
  冻结后未选中的旧积压；可选每日一次治理
        ↓
Trace & Metrics Layer
  选择原因、预计/实际成本、失败与回放
```

### 7.1 Ingestion Layer

职责：

- 创建普通笔记；
- 根据用户选择激活 Book IR；
- 写来源身份；
- 生成首次 eligible window；
- 调用容量感知的首次 due 分配器；
- 保留失败与重试状态。

### 7.2 Cadence Layer

职责：

- 管理 Topic / Extract 的 intentional interval；
- 处理真实进展、停滞、priority 调整；
- 产生下次 due；
- 不关心今天最终是否被选中。

### 7.3 Selection Layer

职责：

- 只读读取 eligible 候选；
- 在 frozen budget 内选择；
- 返回每张入选/未入选原因；
- 不写 block properties。

### 7.4 Frozen Session Layer

职责：

- 保存本次会话约束；
- 保存队列顺序与当前进度；
- 刷新/重开可恢复；
- 消费每日 ledger；
- 防止重复消费。

### 7.5 Overload Layer

职责：

- 只处理冻结队列之外的旧积压；
- 不处理今天新到期内容；
- 不处理 focus / protected 条目；
- 每日最多执行一次；
- 结果可撤销、可审计。

## 8. 统一 Outstanding 的定义

### 8.1 产品定义

本文使用“Outstanding”指：

> 在某个 repo、本地自然日和冻结策略下，所有当前有资格处理、尚未被当日预算消费的学习条目集合。

它不是简单的 `due <= now` 数组，而是由三层组成：

1. **Eligible pool**：今天有资格的全部条目；
2. **Today plan**：在今日总容量中计划处理的条目；
3. **Current session slice**：本次 10/20/30 分钟实际展示的切片。

### 8.2 为什么要分三层

如果直接把 eligible pool 当今日队列：

- backlog 大时用户看到无限任务；
- 无法表达今天只计划完成多少；
- 时间盒之间无法稳定续接。

如果每次时间盒都重新从 eligible pool 选择：

- `dailyLimit` 会重置；
- 刷新顺序变化；
- 同一卡可能重复消费；
- 用户无法信任今日承诺。

### 8.3 建议的会话关系

```text
Daily Outstanding Descriptor
  ├─ 今日预算与已消费 ledger
  ├─ 今日候选选择策略
  ├─ 今日稳定 seed
  └─ 多个 Session Slice
       ├─ 早上 10 分钟
       ├─ 午间 20 分钟
       └─ 晚上继续 10 分钟
```

同一天再次打开时，默认继续今日 plan 的剩余部分，而不是重新生成另一批独立的“每日上限”。

## 9. 冻结描述符设计

> 以下为候选数据形状，尚未决定最终文件名、存储介质或 schema 版本。实施前应评估复用普通 SRS descriptor 基础设施，禁止直接复制出第二套身份与预算逻辑。

### 9.1 候选 `IRDailyDescriptor`

```ts
type IRDailyDescriptor = {
  version: 1;
  repo: string;
  localDate: string;
  policyVersion: string;
  createdAt: number;

  mode: "ir-only" | "mixed";
  scope: {
    sourceBookIds?: number[];
    sourceTopicIds?: number[];
    focusCardId?: number | null;
  };

  budget: {
    totalSeconds: number;
    maxUniqueIR: number | null;
    srsReviewRatio: number;
  };

  constraints: {
    topicMinRatio: number;
    newExtractMaxRatio: number;
    explorationRatio: number;
    sourceRunMax: number;
  };

  seed: string;
  plannedEntryKeys: string[];
  consumedEntryKeys: string[];
  remainingEntryKeys: string[];

  status: "active" | "completed" | "expired" | "invalid";
  lastError?: string | null;
};
```

### 9.2 身份要求

- IR 条目可以用稳定的 IR entry key；
- SRS 条目必须通过现有 `cardIdentity.ts` 产生 `cardKey`；
- 禁止字符串 substring 匹配身份；
- mixed descriptor 中必须区分 `reading` 与 `review` 类型，但不能自造第二套 SRS 身份。

### 9.3 冻结字段

创建后必须冻结：

- repo；
- local date；
- policy version；
- scope；
- IR-only / mixed；
- focus；
- 总时间预算；
- 唯一条目数量预算；
- Topic floor；
- new Extract cap；
- exploration；
- mixed SRS 比例；
- seed。

用户修改全局设置后：

- 已存在 descriptor 默认继续使用旧值；
- 新建下一日或显式“重新计划今天”才使用新值；
- “重新计划”需说明影响，不得静默重排已进行会话。

### 9.4 恢复语义

- 正常刷新：恢复原队列、当前位置和预算；
- 插件重载：从持久 descriptor 恢复；
- 跨本地日：旧 descriptor 标记 expired，创建新日 descriptor；
- repo 切换：不得复用另一 repo descriptor；
- descriptor 损坏：显示可诊断错误，不得静默新建并丢弃旧进度。

## 10. 真正的每日额度

### 10.1 额度定义

建议把当前 `dailyLimit` 拆成两个概念：

1. **每日唯一 IR 条目上限**：跨会话扣减；
2. **单次会话安全上限**：防止异常大数组和 UI 过载。

名称需要在设置 Review 时最终确定；不得继续让一个字段同时表达两种含义。

### 10.2 消费条件

建议只有以下确认成功的动作消费一个 IR unique slot：

- 下一篇成功写回；
- 推后成功写回；
- 完成成功；
- Extract 转 Item 成功且策略结束 Extract；
- 其他明确定义为“本日已处理”的成功动作。

以下不消费：

- 仅打开；
- 只滚动；
- 读取失败；
- 写入失败；
- 对话框取消；
- 同一卡当天已消费后再次打开；
- 仅创建 Extract 而父 Topic 的 next 尚未成功。

“挖空但保留 Extract”是否消费当前 Extract 的当日 slot，需要产品确认。推荐：**消费本日 unique slot，但不结束其长期 IR 生命周期**。

### 10.3 去重

每日 ledger 至少按：

```text
repo + localDate + stableEntryKey
```

去重。

同一卡当天多次进入不同时间盒，只消费一次；但实际动作仍应进入事件日志，用于成本与行为分析。

### 10.4 额度用完

当日 IR 额度用完后：

- 当前卡可以完成；
- 不再补入新的 IR 条目；
- mixed 会话是否继续剩余合法 SRS 取决于 SRS 自己的 remaining budget；
- UI 显示“今天的阅读计划已完成”，而不是“资料库没有内容”。

## 11. 容量感知的首次排期

### 11.1 目标

新章节和新 Extract 的首次 due 不只由自身公式决定，还应考虑允许窗口内的未来预计负载。

### 11.2 Future Load Calendar

建议维护一个可重建的派生视图：

```text
repo + localDate
  -> scheduledTopicSeconds
  -> scheduledExtractSeconds
  -> expectedSrsSeconds（若有可靠接口）
  -> sourceCounts
  -> highPriorityCount
  -> totalEstimatedSeconds
```

它可以是索引/缓存，不应成为唯一事实来源。损坏时可从真实 block properties 和日志有界重建。

### 11.3 负载单位

优先使用预计秒数：

- Topic：沿用现有成本基线和校准；
- Extract：沿用现有成本基线和校准；
- SRS：使用普通复习侧已有或可复用的成本估计；
- 数据不足时使用保守默认值。

张数仅用于：

- 硬 cap；
- 防止单日同类条目过多；
- 诊断展示。

### 11.4 Eligibility Window

先根据内容类型、重要性和用户意图得到允许窗口，再在窗口内选择较低负载日期。

候选示意，非生产参数：

| 条目 | 重要性 | 最早 | 最晚 |
| --- | --- | ---: | ---: |
| 新 Topic | 高 | 1 天 | 4 天 |
| 新 Topic | 中 | 3 天 | 14 天 |
| 新 Topic | 低 | 7 天 | 30 天 |
| 新 Extract | 高 | 1 天 | 5 天 |
| 新 Extract | 中 | 3 天 | 14 天 |
| 新 Extract | 低 | 5 天 | 21 天 |

这些数值只用于说明窗口概念，正式参数必须经 shadow 数据校准。

### 11.5 日期选择评分

候选公式：

```text
dateCost(day) =
  normalizedExpectedSeconds(day)
  + sameSourcePenalty(day)
  + sameTypePenalty(day)
  + highPriorityCrowdingPenalty(day)
  + distancePreference(day)
  + boundedStableNoise(day)
```

选择最低 cost 日期。

约束：

- 不排到窗口外；
- 高重要性不能仅因未来日历较满而无限推远；
- 同源 penalty 只用于分散，不能破坏顺序 Book IR 的解锁语义；
- stable noise 只处理近似并列；
- 失败时保留可解释 fallback，不得静默丢失新条目。

### 11.6 分散 Book IR

当前 `distributed` 按 `totalDays` 均匀铺开。目标算法建议：

1. 保留章节顺序作为弱偏好，不是强锁定；
2. 为每章生成由 `totalDays` 约束的窗口；
3. 按章节顺序逐个放置到低负载日期；
4. 对同一本书设置每日/连续来源上限；
5. 保证选中章节都有首次 due；
6. 初始化部分失败时保留 plan 与失败项，可重试；
7. 重试时不重新扰动已成功章节的 due。

### 11.7 顺序 Book IR

顺序模式保持：

- 仅当前章 active；
- 下一章由完成动作解锁；
- `today` / `tomorrow` 是明确用户意图，优先于全局均衡；
- Future Load Calendar 可以提示负载，但不得把用户选择的 today 偷改为数日后；
- SAC 继续负责当前章 cadence。

## 12. 多日多批 Extract 的全局错峰

### 12.1 当前局部机制保留

现有 sibling queue delay 有明确价值，应继续保留为局部先验：

- 同源；
- 同自然日；
- 创建顺序；
- due-only；
- 有界 BFS；
- 截断可见。

### 12.2 新增全局放置层

建议流程：

```text
创建 Extract
  -> 确认 sourceTopicId / sourceBookId
  -> 计算基础 interval
  -> 计算同源 sibling offset
  -> 形成允许窗口
  -> 查询 Future Load Calendar
  -> 选择窗口内低负载 due
  -> 写 interval（不含临时分散）与 due
  -> 更新派生负载索引
```

### 12.3 同源密度约束

建议至少控制：

- 同一 Topic 同一天首次到期的 Extract 数量；
- 同一本书同一天首次到期的 Extract 数量；
- 单次会话中连续同源条目数量；
- 同源新 Extract 在最终队列中的占比。

### 12.4 示例：连续三天每天 20 个摘录

目标结果不是严格均匀，也不是简单按创建批次形成三条平行线，而是：

- 先满足各自最早/最晚窗口；
- 高重要性更靠前；
- 已有负载较高的日期少放；
- 同源条目尽量分开；
- 每天仍保留少量变化；
- 任何未放置或写入失败都可见。

验收时应检查：

- 每日预计秒数的 P95；
- 峰值/均值比；
- 同源同日密度；
- 窗口越界数；
- 高重要性过晚数；
- 失败与 fallback 数。

## 13. 今日候选评分

### 13.1 Eligibility

先确定资格：

```text
eligible =
  active
  && validState
  && dueDay <= localToday
  && withinFrozenScope
  && notConsumedToday
```

### 13.2 候选 score

建议初始形状：

```text
daysOverdue = max(0, localToday - dueDay)

score =
  priority
  + boundedOverdueBonus(daysOverdue)
  + boundedStarvationBonus(eligibleSkipCount)
  + progressUrgency
  + sourceDiversityAdjustment
  + boundedExplorationNoise
```

### 13.3 Priority

- 继续使用 `ir.priority` 0..100，越大越重要；
- priority 是主导项；
- UI 继续称“重要性”；
- 不从 `#card.priority` 读取调度真相。

### 13.4 Overdue bonus

要求：

- 随逾期天数单调增长；
- 有上限；
- 逾期 30 天应高于逾期 1 天；
- 低 priority 极端逾期不能无限压过高 priority protected 条目。

可从 `min(cap, a * ln(1 + daysOverdue))` 一类有界函数开始 shadow 比较。

### 13.5 Starvation bonus

`eligibleSkipCount` 的定义建议为：

> 某条目在冻结 scope 内有资格，但因预算/约束未入选的不同本地日次数。

不得用以下字段冒充：

- `postponeCount`：它表示真实推后；
- `readCount`：它表示已处理；
- 当前会话中的数组位置。

starvation bonus 必须有上限，并且只有在可靠 selection trace 落盘后才能启用。

### 13.6 Progress urgency

候选信号：

- 顺序 active 章接近完成；
- Topic 已开始且有断点；
- Extract 已 refined / item candidate；
- 长期停滞。

第一版建议只 shadow，不直接影响生产排序，避免把不可靠的断点形状当作强价值信号。

### 13.7 Source diversity

目的不是平均主义，而是避免短时间被单一来源占满。

建议作为装配约束，而不是永久修改 block priority：

- 同 source 连续上限，例如 2；
- 无替代候选时允许放宽；
- 顺序书当前章不因 diversity 被永久排除；
- focus 不受该限制；
- 放宽时写入 diagnostics。

### 13.8 Exploration

- 只从非保护区候选中选择；
- 噪声幅度有界；
- 同 descriptor 可复现；
- 不能破坏时间预算、Topic floor、新 Extract cap、SRS budget；
- 小队列达到阈值时至少尝试一个探索槽；
- 无合法替换时保持原队列并记录原因。

## 14. 最终选择约束优先级

建议明确为：

```text
1. 数据有效、身份有效、eligibility
2. 用户 focus
3. frozen scope
4. 当日剩余 IR / SRS 权威预算
5. 时间预算与单次安全 cap
6. 高重要性旧积压保护
7. Source Topic 最低曝光
8. 新 Extract 最终最大比例
9. 来源连续上限 / 多样性
10. 普通 score 填充
11. 有界探索
12. 最终约束复核与 diagnostics
```

### 14.1 冲突原则

- focus 可独占一个会话，即使自身预计成本超过预算；
- SRS 权威 budget 不可被 mixed 比例绕过；
- 高优 protected 条目可能导致 Topic floor 无法满足，但必须返回原因；
- 只有新 Extract 候选时可软化 new Extract cap，但必须诊断；
- 来源多样性无替代候选时可软化；
- 任何软化都不能伪装成约束已满足。

### 14.2 Selection Result

目标返回值应包含：

```ts
type UnifiedSelectionResult = {
  entries: UnifiedEntry[];
  totalEstimatedSeconds: number;
  budgetSeconds: number;
  diagnostics: SelectionDiagnostic[];
  reasonsByEntryKey: Record<string, SelectionReason>;
  unselectedOldBacklogKeys: string[];
};
```

该形状是候选设计，不代表必须创建同名导出。

## 15. IR 与 SRS 的统一方式

### 15.1 统一的是注意力流，不是调度算法

- IR 条目保留 IR cadence；
- SRS 卡保留 FSRS due、state、rating 和日志；
- Unified Queue 只在两侧各自给出的合法候选和剩余预算内交织。

### 15.2 Mixed SRS 必须复用

- frozen review descriptor；
- frozen scope；
- 今日 new/review remaining；
- due 顺序；
- `cardKey`；
- 已处理日志去重；
- grading gate；
- enqueue/flush 失败语义。

不得只调用 `collectReviewCards` 后自行重新定义一套弱预算。

### 15.3 三种比例必须拆开

| 概念 | 含义 |
| --- | --- |
| Topic floor | 纯 IR 中 Source Topic 的最低曝光 |
| New Extract cap | 纯 IR 中新 Extract 的最大比例 |
| SRS ratio | Unified Queue 中普通 SRS 的目标时间比例 |

它们不能共享一个含糊的“Topic 比例”设置。

### 15.4 Item 新制后的会话语义

默认建议：

- 本次会话中新制的 Item 不动态加入当前 frozen queue；
- 它由 FSRS 初始化并在未来合法时进入复习流；
- 当前 Extract 按 `keep_extract` 继续留在 IR；
- 若未来支持动态补入，也必须遵守原 frozen SRS budget 与身份去重。

## 16. Cadence 与动态 A-Factor

详细算法候选见 [`记忆算法优化.md`](记忆算法优化.md)。本文只规定它与统一队列的边界。

### 16.1 第一原则

统一 Outstanding 不能依赖动态 A-Factor 才成立。应先完成：

- 真正每日额度；
- frozen descriptor；
- selection trace；
- 全局首次排期；
- 统一预算。

之后再替换普通 Topic/Extract 的固定增长倍率。

### 16.2 候选字段

第一版自适应 cadence 可仅新增：

- `ir.aFactor`

是否需要 `ir.scheduleVersion` 应在实现 Review 后决定，不应提前创造多个属性别名。

### 16.3 Lazy migration

- 旧 Topic 缺 AF → 视为 1.25；
- 旧 Extract 缺 AF → 视为 1.35；
- 不批量改旧 due；
- 不批量改旧 interval；
- 下一次真实用户动作时写入；
- 写后 invalidate IR cache；
- feature flag 关闭时恢复固定倍率读取。

### 16.4 进展语义

- 有真实进展：平滑更新 AF，增长 intentional interval；
- 无进展：不把它视为成功阅读；
- postpone：不更新 AF；
- priority：主要影响 selection，弱影响下一次 target AF；
- sibling delay：永不进入 AF 复利；
- 顺序 active 章继续 SAC，不被普通 Topic AF 覆盖。

### 16.5 为什么必须先 Shadow

如果同时上线：

- 新首次排期；
- 新 selection score；
- 新 AF；
- 自动 postpone；

一旦体验变化，无法判断是哪一层造成。动态 AF 应先记录建议值，不改真实 due。

## 17. Overload / Auto-postpone

### 17.1 目标

系统主动治理低价值旧积压，但不把“打开面板”变成不可见的数据修改。

### 17.2 推荐执行顺序

```text
收集 eligible
  -> 冻结 daily/session descriptor
  -> 冻结 focus
  -> 生成最终今日/本次队列
  -> 计算未入选的旧积压
  -> 检查今日是否已经治理
  -> 可选执行 auto-postpone
  -> 持久化实际结果和撤销快照
```

### 17.3 可推迟集合

必须同时满足：

- `dueDay < localToday`；
- 未进入 frozen 今日计划；
- 非 focus；
- 非 protected；
- 当前 scope 内；
- 未超过 postpone protection；
- 当前日尚未被本批处理；
- 块状态和身份读取成功。

### 17.4 写入语义

- 只移动 `due`；
- 保留 `intervalDays`；
- 保留 `position`；
- 不更新 AF；
- 不增加 `readCount`；
- 更新 `postponeCount`、`lastAction=autoPostpone`、batch id；
- 每次 property write 后 invalidate 对应 cache；
- 返回真实 success / failed / rollbackFailed。

### 17.5 每日一次

键至少包括：

```text
repo + localDate + policyVersion
```

不能只使用进程内 Map。插件重载后仍必须知道当天是否已执行。

### 17.6 用户控制

建议分三档：

- 关闭：从不自动治理，只显示诊断；
- 建议：给出“一键治理”摘要；
- 自动：每日首次创建 frozen plan 后执行一次。

默认值需经真实数据和用户 Review 决定。自动模式必须提供：

- 本次推后多少；
- 保留多少；
- 失败多少；
- 撤销入口；
- 失败后可重试状态。

## 18. 日志、Trace 与可解释性

### 18.1 为什么必须先有 Trace

算法优化不能只凭“感觉队列更顺”。需要知道：

- 为什么这张被选中；
- 为什么那张没被选中；
- 预计花多久；
- 实际花多久；
- 是否多次 eligible 却未入选；
- 是否因 budget、比例、来源限制或读取失败被排除。

### 18.2 候选 Selection Trace

```ts
type IRSelectionTrace = {
  repo: string;
  localDate: string;
  descriptorId: string;
  policyVersion: string;
  entryKey: string;
  cardType: "topic" | "extracts";
  due: number;
  priority: number;
  daysOverdue: number;
  estimatedCostSeconds: number;
  scoreTerms: {
    priority: number;
    overdue: number;
    starvation: number;
    progress: number;
    diversity: number;
    exploration: number;
  };
  selected: boolean;
  reason: string;
};
```

候选形状不代表必须逐条写 block properties。优先使用本地调度日志/索引，避免污染正文块。

### 18.3 日志可靠性

- enqueue 成功不等于已确认写盘；
- unload 时在 Orca API 可用阶段 flush；
- flush 失败保持可见和可重试；
- 不因日志失败伪装主业务写入成功；
- 若 trace 失败导致 starvation 统计不可靠，则停用 starvation bonus，而不是使用假默认。

### 18.4 用户可见解释

普通用户无需看到公式。可显示短原因：

- “重要性高”；
- “已等待较久”；
- “继续上次阅读”；
- “保持章节曝光”；
- “今天新摘录配额”；
- “从其他来源穿插”；
- “你手动选择了这篇”。

详情抽屉可展示机器诊断，但默认界面保持低压。

## 19. Future Load Index 的一致性与恢复

### 19.1 权威来源

- block 的真实 `ir.due` / `ir.intervalDays` 是排期事实；
- Future Load Index 是派生加速结构；
- 索引不能反过来覆盖 block 真相。

### 19.2 更新时机

以下成功写入后更新派生负载：

- 新 Topic 初始化；
- 新 Extract 初始化；
- 下一篇更新 due；
- priority 导致 due 变化；
- 手动 postpone；
- auto-postpone；
- 完成/移出 IR；
- Book IR progression；
- retry / rollback。

### 19.3 失败顺序

建议：

1. 先完成权威 block write；
2. invalidate cache；
3. 更新派生索引；
4. 索引更新失败时记录 dirty/rebuild needed；
5. 不把 block write 回滚成失败，除非业务事务明确要求原子性。

### 19.4 有界重建

- 优先按 IR index IDs 读取；
- 有限并发；
- 分页/分批；
- 不对整个 repository 发无界 `Promise.all`；
- 可按日期窗口重建，不必每次计算无限未来；
- 单块失败计入 failedCount，不得静默消失。

## 20. 失败与恢复语义

### 20.1 收集失败

| 情况 | UI | 状态 |
| --- | --- | --- |
| backend 查询失败 | 数据读取失败 | 不生成伪空队列 |
| 部分块读取失败 | 显示部分失败警告 | 成功候选可用，失败 ID 可重试 |
| descriptor 读取失败 | 会话恢复失败 | 不静默创建新 descriptor 覆盖旧状态 |
| 索引失效 | 回退权威读取 | 标记重建，不把索引当真相 |

### 20.2 排期失败

- 新条目 block 已创建但 due 写入失败：保留显式 partial 状态；
- Book IR 部分初始化失败：保留 plan 和失败章节；
- progression 失败：保持当前章或既有可重试检查点；
- 不把写失败条目计入 Future Load Calendar；
- 不把计划数量报告为成功数量。

### 20.3 日额度失败

- 动作业务写成功、ledger 写失败：必须显示“处理成功但今日进度记录失败”；
- 保留 retryable ledger event；
- 重新打开时不得简单假设未消费并重复扣减；
- 需要幂等 event id。

### 20.4 Overload 失败

- 中途失败按现有快照/回滚原则处理；
- rollback 失败列出 block IDs；
- 每日执行标记不能在全批失败时伪装完成；
- 部分成功需记录可重试剩余集合。

## 21. 设置与默认体验

### 21.1 普通用户只需看到

- 每日阅读时长；
- 只读 / 混合；
- 重要性；
- 是否允许自动治理旧积压。

### 21.2 高级设置候选

| 设置 | 说明 | 是否第一期开放 |
| --- | --- | --- |
| 每日时间预算 | 今日计划总分钟 | 是 |
| 单次会话时长 | 10/20/30 分钟切片 | 是 |
| 每日唯一 IR 上限 | 安全数量 cap | 可选 |

| SRS 混合比例 | mixed 目标时间比例 | 保留 20/30/40 |
| 自动积压治理 | 关/建议/自动 | Review 后 |

- “稍后再读”，不用“A-Factor”；
- “已等待较久”，不用“starvation bonus”；
- “今天计划已完成”，不用“eligible pool exhausted”。

## 22. 迁移与兼容

### 22.1 不批量重写旧 due

上线任何新算法时：

- 旧条目保持原 due；
- 新算法从下一次真实动作或新条目创建起生效；
- Future Load Index 初次建立只读取，不重排；
- 用户显式选择“重新平衡未来计划”前，不批量移动现有 due。

### 22.2 旧设置

当前 `dailyLimit` 语义是单次硬上限。迁移到真实每日额度时必须：

- 明确 schema / UI 文案变化；
- 保留合理默认；
- 如果拆成两个 setting，定义旧值如何映射；
- 不静默把旧 30 解释成每日 30 且取消单次安全 cap。

### 22.3 旧 IR 数据

- 缺 sourceTopicId 的 legacy Extract 继续使用现有兼容读取；
- 新创建路径必须写耐久 sourceTopicId/sourceBookId；
- 缺 AF 时 lazy fallback；
- 缺 descriptor 不代表卡损坏，只代表尚未创建新日计划；
- 不删除未知 `ir.*` 字段。

### 22.4 Feature Flags

候选能力开关：

- trace only；
- future-load scheduling shadow；
- adaptive selection shadow；
- adaptive interval；
- auto overload；
- unified mixed budget。

最终 setting key / feature flag 名称需实现时 Review，本文不把概念名当成已经存在的 schema。

## 23. 分阶段开发计划

### Phase 0：运行时基线与产品口径

#### 目标

在不改行为前，确认真实 Orca 数据形状和当前负载。

#### 工作

- 采样 Topic、Extract、Book IR、hybrid Cloze；
- 对比 `orca.state` 与 backend；
- 统计 due 分布、priority、来源、未来每日预计量；
- 记录同源同日 Extract 密度；
- 明确“今日额度消费动作”；
- 决定 mixed 是否作为默认入口。

#### 退出条件

- 有可复制的只读运行时样本；
- 现有属性 shape 与本地文档一致；
- 确认不需要发明新的 Orca API；
- 产品决策表签字/确认。

---

### Phase 1：Selection Trace 与只读 Shadow

#### 目标

先解释当前 v1 为什么选中，再计算新策略建议，不改变真实顺序。

#### 工作

- frozen policy input snapshot；
- v1 selection reason；
- v2 shadow score；
- 未入选原因；
- 预计成本与实际停留；
- 日志 enqueue/flush/retry；
- replay 工具或纯函数测试夹具。

#### 退出条件

- 相同 descriptor replay 结果一致；
- trace flush 失败可见；
- 可比较 v1/v2 队列差异；
- 不写 block due；
- 连续积累至少 2～4 周真实样本后再评估下一阶段参数。

---

### Phase 2：Frozen Daily Descriptor + 真正每日额度

#### 目标

建立可信的“今天”。

#### 工作

- 持久 daily descriptor；
- session slice 恢复；
- unique consumption ledger；
- repo / local day 隔离；
- `dailyLimit` 语义拆分/迁移；
- 刷新、重开、跨面板去重；
- 今日完成与资料库空状态区分。

#### 退出条件

- 同日重开不重置额度；
- 同一条目一天只消费一次；
- 失败动作不消费；
- descriptor 损坏不静默覆盖；
- 跨 repo / 跨日不混用；
- 自动化测试覆盖恢复与幂等。

---

### Phase 3：Mixed SRS 权威预算对齐

#### 目标

统一注意力流，同时保留 FSRS / SRS 权威职责。

#### 工作

- 复用普通 SRS frozen descriptor / scope；
- 复用 new/review remaining；
- 统一 cardKey；
- mixed selection 不重复消费；
- grading log 与 descriptor ledger 对齐；
- 动态补入仍受 frozen budget（若本期实现）。

#### 退出条件

- mixed 不超过普通 SRS remaining；
- 同一卡不被两个会话重复加入；
- grading 失败不消费；
- IR-only 行为不受影响；
- 普通独立 SRS 会话回归通过。

---

### Phase 4：Future Load Calendar（Shadow）

#### 目标

计算新章节/Extract 如果容量感知会被放到哪一天，但暂不改真实 due。

#### 工作

- 派生未来负载索引；
- 有界重建；
- eligibility window；
- date cost；
- 同源密度；
- Book distributed shadow placement；
- 多日多批 Extract 模拟。

#### 退出条件

- 索引可由权威数据重建；
- 峰值/均值优于当前算法；
- 高重要性不越过最晚窗口；
- 结果稳定可复现；
- 读取失败有 failedCount；
- 1 万条规模不使用无界并发。

---

### Phase 5：启用容量感知首次排期

#### 目标

只对新创建的章节/Extract 使用 Future Load Calendar。

#### 工作

- 新 Book distributed placement；
- 新 Extract placement；
- 权威写后更新索引；
- partial / retry；
- feature flag 与回滚；
- 不重排旧 due。

#### 退出条件

- 新条目窗口合法；
- 同源同日峰值下降；
- 索引失败有安全 fallback；
- 重试不扰动已成功条目；
- 写后 cache invalidation 正确；
- Orca backend 复读确认通过。

---

### Phase 6：Adaptive Selection

#### 目标

启用有界 overdue aging、可靠 starvation 与来源多样性。

#### 工作

- score terms；
- protected zone；
- bounded overdue；
- trace 派生 eligibleSkipCount；
- source run constraint；
- 最终约束复核；
- 用户可见短原因。

#### 退出条件

- 高重要性在同条件下稳定优先；
- 逾期补偿单调且有上限；
- protected 不被随机越过；
- 低 priority 有 mercy 路径；
- 无可靠日志时 starvation 自动停用；
- 时间预算和比例约束仍成立。

---

### Phase 7：每日一次旧积压治理

#### 目标

让用户无需每天手动整理 backlog，同时保留透明和撤销。

#### 工作

- 关/建议/自动三档；
- 每日持久执行标记；
- 只处理旧积压；
- frozen/focus/protected 排除；
- 批次快照与跨重载撤销；
- 真实 success/failure；
- UI 摘要。

#### 退出条件

- 同日重开不重复推后；
- today due 不被处理；
- focus 不被处理；
- 只改 due；
- 部分失败和 rollbackFailed 可见；
- 关闭后立即停止自动行为。

---

### Phase 8：动态 A-Factor

#### 目标

用逐卡、进展感知的 cadence 替换固定倍率，但不改变统一队列边界。

#### 工作

- `ir.aFactor` lazy migration；
- progress-aware target；
- stall guard；
- priority 弱耦合；
- 顺序 SAC 分流；
- shadow → feature flag → rollout。

#### 退出条件

- 不批量改 due；
- 无进展不增长；
- postpone 不改 AF/interval；
- sibling delay 不复利；
- feature flag 关闭恢复 v1；
- 真实数据指标优于或不劣于 baseline。

## 27. 关键验收场景

### 场景 A：一次导入十章

输入：

- distributed；
- 10 章；
- 中等重要性；
- 30 天窗口；
- 未来已有其他书负载。

期望：

- 每章都有首次 due；
- 不机械叠在已有高负载日；
- 同书不过度连续；
- 不越 30 天约束；
- 重试失败章不改成功章。

### 场景 B：同时导入十本书

期望：

- 不是十本书分别均匀、合起来扎堆；
- 全局预计分钟峰值受控；
- 来源交织；
- 高重要性书保持更近窗口；
- 低重要性书仍有未来安排。

### 场景 C：连续三天每天做二十个 Extract

期望：

- 同源 sibling 局部错峰继续有效；
- 三个 batch 不形成明显重叠尖峰；
- high priority 更早但不全部同日；
- future load 超标时在允许窗口中寻找较空日期；
- 任一失败 Extract 可诊断。

### 场景 D：每日只打开并处理

期望：

- 默认继续当日剩余计划；
- 不要求先看资料库；
- 同日重开额度不重置；
- 时间到可直接离开；
- 次日新 descriptor 正确建立。

### 场景 E：低重要性长期积压

期望：

- priority 仍主导；
- 逾期补偿有上限；
- 可靠 trace 后获得 starvation 补偿；
- 有 mercy/exploration 机会；
- 不会因随机直接越过 protected 高优内容；
- 用户可看到“已等待较久”的原因。

### 场景 F：Extract 制作 Cloze

期望：

- Cloze 成功进入 FSRS；
- `keep_extract` 时 Extract 保留 IR；
- 当日 unique slot 语义一致；
- 新 Item 不绕过 SRS budget 动态挤入；
- 来源追溯保留；
- 失败补偿不破坏原 Extract。

### 场景 G：自动治理

- 只处理冻结后未入选的旧积压；

- 同日不重复；
- 只改 due；
- 摘要与撤销可用；

## 28. 风险与权衡

### 28.1 过度自动化

风险：系统悄悄移动大量 due，用户失去信任。

缓解：

- 会话装配保持只读；
- 自动治理独立、每日一次；
- 关/建议/自动；
- 摘要、批次和撤销。

### 28.2 复杂算法难以解释

风险：score terms 太多，用户感到不可预测。

缓解：

- priority 保持主导；
- bonus 有界；
- 随机幅度小；
- 用户只看短原因；
- 先 shadow。

### 28.3 Future Load Index 漂移

风险：索引与真实 due 不一致。

缓解：

- block properties 为权威；
- dirty 标记；
- 有界重建；
- 写后更新；
- 诊断 mismatch。

### 28.4 长 Topic 被成本模型饿死

风险：如果按 `score / cost` 排序，短 Extract 总是更划算。

缓解：

- 成本只用于装箱，不替代价值；
- Topic floor；
- started Topic / sequential active 保护；
- focus 可独占。

### 28.5 Starvation 数据不可靠

风险：日志丢失后错误补偿。

缓解：

- trace flush 可见；
- 无可靠日志则 bonus=0；
- 不用 postponeCount 冒充。

### 28.6 与普通 SRS 重复建设

风险：为 mixed 再造 scope、budget、identity。

缓解：

- 复用普通 SRS descriptor 和 budget；
- cardKey 只走现有 identity；
- Unified Queue 只做组合层。

### 28.7 参数过多

风险：用户需要理解算法才能使用。

缓解：

- 可靠默认；
- 普通设置只保留时间、模式、重要性和自动治理；
- 数学参数只用于开发/实验。

## 29. 待产品决策

在开始实现前，需要明确：

1. “今日学习”是否默认混合 SRS，还是默认纯阅读？
2. 每日总预算是用户设一个总分钟，还是 IR 与 SRS 各自预算后再组合？
3. `dailyLimit` 是否拆为“每日 unique 上限”和“单次安全 cap”？
4. “挖空但保留 Extract”是否消费该 Extract 的当日 unique slot？
5. 自动积压治理默认是关闭、建议还是自动？
6. 低重要性条目是否需要可配置的最大等待目标，还是只用 bounded starvation？
7. 同一来源连续条目上限默认是 2、3，还是只做 soft penalty？
8. Book distributed 的 `totalDays` 是硬最晚边界还是目标跨度？
9. 用户选择“今天阅读”时是否允许超出当日 unique budget？推荐允许 focus，但该会话可只放 focus。
10. 今日计划是否允许用户显式“重新计划”？如果允许，已消费条目和 frozen SRS scope 如何保留？
11. 是否需要“只阅读 / 只复习 / 混合”三个日常入口，还是一个入口内切换？
12. Future Load Calendar 是否纳入 SRS 预计负担；如果普通 SRS 没有可靠未来成本接口，首期是否只平衡 IR？

## 30. 建议的首轮开发范围

虽然本文描述完整终态，**首轮不应直接实现全部算法**。

推荐第一轮只做：

1. Selection Trace；
2. frozen daily descriptor；
3. 真正跨会话每日 unique ledger；
4. 当前 v1 队列的可解释 reason；
5. v2 selection shadow；
6. Future Load Calendar 只读 shadow；
7. mixed SRS budget 复用方案设计与测试护栏。

明确不在首轮启用：

- 动态 A-Factor；
- 自动移动旧积压；
- 批量重排旧 due；
- starvation bonus 生产排序；
- 新 Future Load due 写入；
- 大量高级设置。

首轮的成功标准不是“算法已经更聪明”，而是：

> **今天的范围和额度可信，选择过程可重放，后续算法有真实数据可以安全比较。**

## 33. 最终定义

本设计完成后的系统不应被描述为“SuperMemo 算法复刻”，而应描述为：

> **Orca 中的统一注意力队列：使用 IR 调度推进阅读材料，使用 FSRS 维护主动回忆，在冻结的每日预算内按重要性、等待时间、来源多样性和容量进行可解释选择。**

它的最终验收不是公式与 SuperMemo 相似，而是用户是否获得以下稳定体验：

- 可以持续导入，不害怕未来爆量；
- 可以读一点就离开，相信现场会保存；
- 可以大量摘录，不必当天立即制卡；
- 相信重要内容会较快回来；
- 相信低重要性内容虽晚但不会无声消失；
- 相信今天的工作量有限且不会因刷新重置；
- 相信错误会被明确报告；
- 每天只需打开“今日学习”，处理当前条目，然后离开。
