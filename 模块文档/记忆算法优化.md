# 记忆算法优化：IR 阅读调度调查与改进方案

> **状态：计划 / 待 Review / 未落地**  
> **调查日期：2026-07-18**  
> **适用范围：Topic、Extract、Book IR、IR 与普通 SRS 的混合队列**  
> **实现真相：始终以当前源码与相关模块文档为准。本文件中的目标公式、字段和参数均是候选设计，不代表现有行为。**

## 0. 决策摘要

本轮调查的结论是：

1. 普通记忆卡继续使用 FSRS，不应改动这一职责边界。
2. Topic 与 Extract 不应直接套用 FSRS。阅读材料的目标不是维持某个可测记忆保持率，而是在有限时间内持续推进、提炼和淘汰材料。
3. 当前 IR 调度可以理解为一个简化的静态 A-Factor：Topic 每次乘 `1.25`，Extract 每次乘 `1.35`。方向合理，但缺少逐卡自适应、进度反馈和完整的过载治理。
4. 推荐升级为 **IR-AF v2 双层调度**：
   - `due / interval / A-Factor` 决定材料何时重新成为候选；
   - `priority + bounded aging + 有界随机探索` 决定候选在今天的先后；
   - 独立的 overload 层只处理确实无法完成的旧积压。
5. 在更换算法前，必须先修复当前队列设置、配额、每日额度和自动推迟语义的偏差。否则无法正确评估新算法。
6. 新算法应先以 shadow mode 运行，记录“新算法会怎样选”，但不改变真实队列。积累真实使用数据后再启用。

一句话概括：**FSRS 负责记忆保持，IR-AF 负责阅读推进，priority queue 负责有限时间内的价值排序。**

## 1. 术语与边界

| 名称 | 当前项目含义 | 调度归属 |
| --- | --- | --- |
| `topic` | 原始文章、普通 Topic、Book IR 章节 | IR 阅读调度 |
| `extracts` | 从 Topic 提取、仍需继续加工的摘录 | IR 阅读调度 |
| `excerpt` | 项目中的另一种普通记忆卡类型，不等于 IR Extract | FSRS |
| Cloze / Direction / List / Choice | 主动回忆卡 | FSRS |
| Extract 转 Cloze | 转换成功前是 IR，成功后清理 IR 排期并进入 FSRS | 分界动作 |
| `priority` | 项目中 `0..100`，数值越大越优先 | 入队排序与阅读节奏的弱耦合信号 |
| `intervalDays` | 最近一次真实阅读动作产生的阅读间隔 | IR 候选生成 |
| `due` | 下一次允许进入候选集的时间 | IR 候选生成 |

特别注意：SuperMemo 使用 `0%` 表示最高优先级，而本项目使用 `100` 表示最高优先级。比较机制时可以类比，不能直接比较数值。

## 3. 当前实现调查

### 3.1 主调用链

时间盒会话的主要链路如下：

```text
IRReadingView
  -> useIRWorkspaceSession.loadReadingQueue
  -> collectIRCardsDetailed
  -> selectQueueWithPolicy
  -> applyAutoPostpone
  -> focus 插队
  -> 可选混入普通 SRS
  -> session.entries 内存快照
```

用户动作链：

```text
下一篇
  -> performNext
  -> markAsRead
  -> 计算 interval / due
  -> saveIRState

推后
  -> performPostpone
  -> postpone
  -> 写回 interval / due / postponeCount

调重要性
  -> performPriorityAdjust
  -> 调整 priority / interval / due
```

主要文件：

- `src/components/incremental-reading/workspace/useIRWorkspaceSession.ts`
- `src/srs/incrementalReadingCollector.ts`
- `src/srs/incremental-reading/irQueuePolicy.ts`
- `src/srs/incremental-reading/irSessionService.ts`
- `src/srs/incremental-reading/irSchedulingMutations.ts`
- `src/srs/incremental-reading/irStatePersistence.ts`

### 3.2 卡种分流

`src/srs/reviewCardFactory.ts` 明确排除 `topic` 与 `extracts`，因此：

- Topic / Extract 使用 IR 自定义排期；
- 普通卡调用 `ts-fsrs@5.2.3`；
- Extract 成功转为 Cloze 后才进入 FSRS；
- 自动 Extract 路径即使写入部分 `srs.*`，也不会因此被普通 SRS 收集器接纳。

这条职责边界应保留。

### 3.3 当前 IR 状态

核心持久化字段：

- `ir.priority`
- `ir.lastRead`
- `ir.readCount`
- `ir.due`
- `ir.intervalDays`
- `ir.postponeCount`
- `ir.stage`
- `ir.lastAction`
- `ir.position`
- `ir.resumeBlockId`
- `ir.breakpoint`
- `ir.autoPostponeBatchId`
- `ir.sourceTopicId`
- Book IR 来源与计划字段

`saveIRState` 写属性后会失效 IR cache，并同步 `#card.priority`。任何新增属性写入也必须保持同样的缓存失效约束。

### 3.4 当前基础间隔

令：

```text
p = round(clamp(priority, 0, 100)) / 100
```

基础间隔：

```text
Topic:   base = round(14 - 12p)  // 约 14d -> 2d
Extract: base = round(7 - 6p)    // 约 7d  -> 1d
```

实现：`src/srs/incrementalReadingScheduler.ts`。

### 3.5 当前分散与兄弟 Extract 延迟

新卡分散：

```text
Topic:   interval = base + r * min(1, 0.5 * base)
Extract: interval = base + r * 0.5 * base + queueDelay
```

非新卡分散：

- Extract：`±min(3, 0.35 * base)`；
- Topic：`±min(2, 0.2 * base)`。

同 Topic 当日第 `index` 个 Extract：

```text
step = min(0.5, max(0.15, base * 0.2))
queueDelay = index * step
```

随机 seed 基于块 ID、日期、卡种和 salt，目标是同一天可复现。

### 3.6 当前后续增长

普通下一篇：

```text
Topic:   next = min(60, interval * 1.25)
Extract: next = min(30, interval * 1.35)
```

这等价于所有 Topic 固定 `AF=1.25`，所有 Extract 固定 `AF=1.35`，再加硬上限。

### 3.7 顺序 Book IR 的 SAC

顺序书当前激活章不走 Topic 的 `×1.25`：

```text
base = 1 + 2 * (1 - priority / 100)
```

- priority `0 / 50 / 100` 约为 `3 / 2 / 1` 天；
- 阅读进度指纹不变时，每次增加约 1 天；
- 最长约 6 天；
- 目标是降低无进展章节持续占用队列，而不是记忆保持。

这套 stall 思路应保留，但应与新的普通 Topic stall guard 统一概念和测试。

### 3.8 当前队列策略

主时间盒策略：

- 候选条件：本地自然日 `due <= 今天`；
- 顺序：priority 降序，然后只区分“逾期/今天”，最后稳定随机；
- 不比较逾期天数；
- 默认 Topic 最低比例 `10%`，至少 1 张；
- 默认新 Extract 最大比例 `20%`；
- 默认探索比例 `5%`；
- 高优先级阈值 `80`；
- 时间成本采用卡种基线，至少 3 个停留样本后使用 `0.4 * 基线 + 0.6 * 历史均值`。

当前 IR `dailyLimit` 实际是每次生成队列的数量上限，不会像普通 SRS 一样从当天已完成日志中扣减。

### 3.9 当前实现值得保留的部分

- IR 与 FSRS 的职责分离；
- 有界并发收集与索引回退；
- 稳定随机而不是每次打开乱跳；
- 时间盒与真实停留时间校准；
- 新 Extract 的首次分散；
- Book IR 的进度停滞保护；
- 过载写入的快照、批次和回滚方向；
- 错误应可见、失败状态可重试的整体原则。

## 4. 实施前必须处理的 P0 问题

以下结论来自静态源码检查。实现 agent 应逐项复核，并在真实 Orca 实例中取得只读证据后再修改涉及属性与 backend 的路径。

### 4.1 设置与真实行为不一致

`enableAutoDefer` 的文案和模块文档表示它只控制“一键溢出推后”按钮，打开面板不会改排期；但 `useIRWorkspaceSession` 在加载时间盒队列时无条件调用 `applyAutoPostpone`。

影响：

- 关闭设置也可能发生 due 写入；
- 打开阅读会话不再是只读选卡；
- 无法把“算法选中差异”和“提前被改期”区分开。

### 4.2 `topicQuotaPercent` 未进入主时间盒队列

该设置只进入旧 collector / 资料库健康条，实际时间盒固定使用 `topicMinRatio=0.1`。

需要先明确两个不同概念：

- Source Topic 与 Extract 在纯 IR 队列中的比例；
- 所有 IR Topics 与普通 SRS Items 在混合队列中的比例。

不能继续用一个含糊的 `topic` 设置同时表达两者。

### 4.3 新 Extract 上限可被绕过

`newExtractMaxRatio` 只约束优先添加阶段；后续价值填充再次遍历所有 Extract，可把新 Extract 加回来。

要求：比例约束必须在最终结果上验证，而不是只约束某一个阶段。

### 4.4 focus 顺序错误

当前先自动推迟未选旧积压，再把 focus 卡插入会话。用户明确点开的卡可能先被改期，然后又出现在本次会话。

要求：focus 必须在任何 overload mutation 之前冻结，并从可推迟集合中排除。

### 4.5 优先级调整路径漂移

主 UI 的 `performPriorityAdjust` 与另一条 `updatePriority` 路径使用不同的 interval clamp。Extract 可能越过 30 天，Topic 也可能出现不一致。

要求：优先级影响间隔的逻辑只能有一个纯函数真相。

### 4.6 首次 `queueDelay` 污染长期 interval

新 Extract 的兄弟错峰被直接写进 `intervalDays`，后续 `×1.35` 会把一次性错峰继续复利。

要求：首次 sibling offset 只改变首次 `due`，不应进入长期学习 interval。

### 4.7 Topic 保底是 best-effort

高优先级阶段可能先占满数量或时间预算，导致 Topic floor 无法满足。

要求：明确约束优先级，并在无法满足时返回原因，而不是静默违约。

### 4.8 探索阶段存在两个偏差

- `floor(queueSize * 0.05)` 使少于 20 张的队列完全没有探索；
- 探索替换后未重新计算总成本，可能超过时间预算。

### 4.9 `dailyLimit` 不是每日额度

处理 30 张后重新打开会话仍可继续取另外 30 张。

要求：每日额度必须从本地当天已完成的唯一 IR 条目中扣减；重复打开同一张不能重复消费额度。

### 4.10 混合 SRS 绕过普通预算

混合模式直接收集普通复习卡，再按时间比例选择：

- 没有复用普通 SRS 的跨会话 daily budget；
- 没有严格复用普通队列的 due 顺序；
- 新会话可能重复消费同一日额度。

### 4.11 IR 会话没有持久化冻结描述符

普通 SRS 会话有 frozen descriptor / scope / budget；IR 只有 React 内存快照，刷新会重新收集。

要求：同一 IR 会话的 scope、日期、模式、预算、seed 和配额必须可说明且稳定。

### 4.12 日期 seed 使用 UTC

主队列使用 `toISOString().slice(0, 10)`，而 due 与每日额度使用本地日。东八区 `00:00..07:59` 会继续使用前一天 UTC seed。

要求：所有“每日”语义统一使用仓库已有本地日 helper。

### 4.13 创建与缓存风险

- 自动 Extract 在失效 IR cache 前调用 `ensureIRState / updatePriority`，可能按旧卡种初始化；
- 手动 Extract 写 `ir.sourceTopicId` 后没有失效对应 IR cache；
- 继承 Topic priority 的部分路径只读 `orca.state`，失败后回退 50，可能隐藏真实读取失败。

涉及属性写入时必须遵守：写入后失效匹配缓存，再允许后续读取依赖该值。

### 4.14 失败结果可能被包装成成功

- 手动溢出推后内部使用 `allSettled`，失败只 warn，外层仍可能返回原计划数量；
- 新鲜索引路径中单块读取失败，在失效比例未达到阈值时可能直接消失且不进入 `failedCount`。

要求：实际成功数、失败数和失败 ID 必须来自写入结果，不得来自计划数量。

## 5. SuperMemo 调度调研

### 5.1 官方可确认事实

#### Extract 是 Topic，不是 Item

SuperMemo 将 Extract 定义为从大文章中取出的片段，它以独立 Topic 身份进入 learning process，并作为源文章的 child。

Topic 用于被动阅读和上下文；Item 才用于 active recall 与长期记忆。

#### Topic 与 Item 的算法相似但不相同

SuperMemo 15 旧帮助明确说明：

> The algorithms that determine the timing of repetitions of question-and-answer material and reviewing reading material are analogous but not identical.

Item 使用复杂的记忆算法；Topic 的 inter-review interval 算法更简单，也允许用户直接控制。

#### Topic 可确认的公式

```text
nextInterval = currentInterval * topicAFactor
```

官方例子：`A-Factor=2` 时，间隔每次翻倍。

Forgetting index 不用于 Topic；当 Topic 产生 Item 后，forgetting index 才对 Item 生效。

#### A-Factor 是启发式更新

官方公开影响因素包括：

- 文本长度；
- 文本如何被处理；
- 是否被 postpone；
- 是否被 advance；
- 其他未公开因素。

旧帮助给过直觉示例：

- 长文本可能使用约 `AF=1.1`；
- 短 Extract 可能使用约 `AF=1.8`。

这些只是示例，不是固定常量。

#### priority 与 interval 分工

- interval / due date 决定某天是否进入 outstanding；
- priority 决定当天先处理谁，以及过载时保护谁；
- 所有 Topic 与 Item 共用相对 priority queue；
- 默认会 auto-sort，但允许一定程度的 randomization；
- 用户可设置 Topics 在混合学习流中的比例。

官方经验起点是每个 Topic 至少穿插约 4 个 Items。它是负担管理建议，不是闭源调度公式。

#### Auto-postpone

SuperMemo 的 auto-postpone：

- 只处理此前日期遗留的 outstanding；
- 今天刚到期的材料不受影响；
- 可按 priority、类型、interval、A-Factor、postpone count 等保护；
- delay factor 可乘已有 interval，并有 min/max delay；
- 官方明确警告 postpone 会偏离目标 retention。

### 5.2 可复原的公开流水线

```text
创建或真实处理 Topic
  -> 基于 Topic A-Factor 生成下一 interval / due
  -> 到期后进入 outstanding
  -> 统一 priority queue
  -> topic/item proportion + priority sort + configurable randomization
  -> 用户处理
  -> 未完成的旧积压按保护条件 postpone
```

### 5.3 仍然闭源或未知

公开资料不能确认：

- 普通 Remember Extract 的自动 first interval；
- 新 Topic / Extract 的 AF 精确初值；
- Extract 从父 Topic 继承 priority 的精确换算；
- length / processing / postpone / advance 的权重和更新条件；
- priority 改动与 AF、interval 的精确双向映射；
- topic/item 混排概率、randomization 分布和 tie-break；
- 不同 SuperMemo 版本的参数上下界和默认配置。

因此可以借鉴的是架构与公开规则，不能声称复刻 SuperMemo 算法。

### 5.4 社区旁证

社区讨论较一致地观察到：

- 新 Extract 会被分布到未来不同 NextRep 日期；
- Next repetition 沿当天 outstanding queue 前进；
- 创建 Extract 后不应立即把所有内容一次性制卡，而应让它重新进入阅读流；
- whether it appears today 取决于 scheduled date、outstanding 和 auto-postpone。

这些观察支持“双层模型”，但没有提供可信的闭源公式。

## 6. 当前实现与 SuperMemo 对比

| 维度 | 当前项目 | SuperMemo 公开机制 | 改进方向 |
| --- | --- | --- | --- |
| 普通 Item | FSRS | SM-17/18 等记忆模型 | 继续 FSRS |
| Topic interval | 固定 `×1.25` | 每 Topic A-Factor | 动态、可解释的逐卡 AF |
| Extract interval | 固定 `×1.35` | Extract 仍是 Topic，通常因短小获得更大 AF | 动态 AF + 快速加工漏斗 |
| priority | 同时影响初始 interval、顺序、推迟 | 与 AF 分离但存在启发式耦合 | priority 主导价值，弱影响 cadence |
| overdue | 只区分逾期/今天 | outstanding + priority / postpone | 有界 overdue aging |
| randomization | 同分组稳定随机 + 5% 探索 | 可配置 priority-biased randomization | 优先级窗口内有界随机 |
| 类型比例 | Topic floor、新 Extract cap、混合比例 | Topic/Item proportion | 明确拆分三种比例语义 |
| overload | 当前时间盒路径会自动写入，设置语义冲突 | 只处理旧积压，可配置保护条件 | 每日一次、队列冻结后、真实结果可见 |
| 时间预算 | 有估算与历史校准 | 公开文档以配额和过载为主 | 保留，这是现有优势 |
| 进度反馈 | Book SAC 有；普通 Topic 很弱 | 长度和处理行为参与 AF | progress-aware AF + stall guard |

## 7. 开源路线调研

截至 2026-07-18，没有发现成熟、高星、可直接替代 SuperMemo 阅读调度的算法库。可借鉴的是局部机制。

| 项目 | Stars / 活跃度 / 许可 | 真实阅读调度 | 可借鉴点 | 限制 |
| --- | --- | --- | --- | --- |
| `bjsi/incremental-everything` | 74；2026-07-14；MIT | 是 | 相对/绝对 priority、继承、Structure/Exploration、卡片阅读比例、指数或渐近 interval | RemNote 生态，不能整库移植 |
| `campfirium/foliole` | 80；2026-07-18；Apache-2.0 | 是 | 完整 IR 产品、本地数据、集成 `ts-fsrs` | 使用 FSRS 调阅读只证明可实现，不证明科学有效 |
| `mochar/logseq-incremental-blocks` | 54；2025-04-18；AGPL-3.0 | 是 | due 先筛选，`UUID+date` 稳定 Beta 随机 priority，interval 倍增 | 维护暂停；停留时长反馈含义不可靠 |
| `kja140/incremental-reading` | 3；2026-07-13；MIT | 是 | progress-aware A-Factor、stall guard、Topic/Card 分离、Mercy | 很新，采用量低 |

| `open-spaced-repetition/fsrs4anki` | 4016；2026-03-20；MIT | 否 | 成熟普通记忆调度 | 无文章选择、阅读点、Extract、priority overflow |
| `open-spaced-repetition/ts-fsrs` | 716；2026-07-16；MIT | 否 | 当前项目已使用的通用 FSRS | 不是 IR queue |
| `supermemo/SuperMemoAssistant` | 212；2023-01-07；MIT | 否 | 扩展已安装的闭源 SuperMemo | 不包含调度器 |

定向搜索没有找到可信的完整 SM-17/18 开源实现，更没有覆盖 IR 选择层的实现。`fsrs-vs-sm17` 是比较 notebook / 数据集，不是 SM-17 实现。

### 7.1 应吸收的开源机制

- due 资格与 priority 排序分层；
- 每日确定性随机，跨天可变化、同日可复现；
- progress-aware A-Factor；
- stall guard；
- Soon / Normal / Later 的显式节奏纠偏；
- Topic 与 Item 混合比例；
- Mercy / spread overdue；
- 阅读与记忆调度职责分离。

### 7.2 不应直接照搬

- 把停留时长当作强价值信号；
- 把“读过一段”伪装成 FSRS 的 Remember；
- 无保护区的全局随机；
- 未经真实数据校准的复杂 Beta / bandit 更新；
- 直接迁移停更插件的代码或存在许可疑点的实现。

## 8. IR-AF v2 设计目标

### 8.1 目标

- 高价值材料保持较高可见度；
- 长文章可以小步推进，而不是被短 Extract 永久淹没；
- 低价值材料允许自然淡出，但不会因排序细节永久饥饿；
- 新 Extract 不在少数日期结块；
- 无进展不会继续把 interval 当作成功阅读来增长；
- overload 行为可预测、可撤销、可解释；
- 时间盒尽量接近用户选择的分钟数；
- 每次选中都能解释原因；
- 旧数据迁移不批量重写 due。

### 8.2 非目标

- 不复刻闭源 SuperMemo；
- 不用 FSRS 预测文章阅读时间；
- 不用停留时间推断“记住了”；
- 不在第一版引入机器学习模型；
- 不自动修改正文、priority 或卡片类型；
- 不为了算法统一而删除 Book IR 的顺序推进语义。

### 8.3 不变量

- 普通卡仍经 FSRS validated settings path，21 个 weights 约束不变；
- Topic / Extract 不进入普通 SRS collector；
- 每次块属性写入后必须失效匹配 cache；
- 错误必须可见，不能用默认 50、空数组或计划数量伪装成功；
- 收集和子块展开必须继续有并发、深度和数量上限；
- 跨队列身份需要 `cardKey` 时，只能经 `src/srs/cardIdentity.ts`；
- 会话 descriptor、scope、每日预算与 seed 冻结；
- 所有每日边界使用本地日，不使用 UTC 字符串截断。

## 9. IR-AF v2 总体架构

建议分为三个纯逻辑层：

```text
Cadence Layer
  输入：上次真实处理、interval、AF、priority、进度
  输出：下次 interval / due

Selection Layer
  输入：今天到期候选、priority、overdue、历史跳过、成本、配额
  输出：冻结的时间盒队列 + 每张入选原因

Overload Layer
  输入：冻结后未选中的旧积压
  输出：可选的 due 推迟写入结果
```

三个层不能互相偷改语义：

- Selection 不得把未选中直接等同于已成功 postpone；
- Postpone 不得伪造一次真实阅读并增长长期 interval；
- priority 调整不得重置阅读进度；
- sibling dispersal 不得进入长期 AF 复利。

## 10. 候选数据模型

### 10.1 最小新增持久字段

第一版只建议新增一个必要字段：

- `ir.aFactor`：Topic / Extract 当前阅读间隔增长因子。

候选、需 Review 后才能落地：

- `ir.scheduleVersion`：仅在确实需要显式迁移版本时添加；否则可用 `ir.aFactor` 是否存在进行 lazy migration。

不建议第一版新增：

- 单独的 `deferUntil`：可以通过只改 `ir.due`、不改 `ir.intervalDays` 表达；
- 大量 block 级 score 字段：score 应在会话中派生；
- 把停留时长、selection reason 全写回块属性：应进入本地调度日志。

最终属性名必须经过项目 property namespace review；不得同时创建多个别名。

### 10.2 派生状态

以下值按会话计算，不必写入每个块：

- `daysOverdue`
- `estimatedCostSeconds`
- `progressAdvanced`
- `remainingRatio`（可获得时）
- `eligibleSkipCount`（有日志后）
- `scoreTerms`
- `selectionReason`
- `isProtected`

## 11. 动态 A-Factor

### 11.1 迁移初始化

为避免旧排期突然改变：

```text
existing Topic without ir.aFactor   -> 1.25
existing Extract without ir.aFactor -> 1.35
```

迁移必须 lazy：

- 不批量改现有 `due`；
- 不批量改现有 `intervalDays`；
- 在下一次真实用户动作时写入 AF；
- 写入后失效 IR cache；
- 回滚时忽略 `ir.aFactor` 即可恢复旧倍率逻辑。

### 11.2 候选目标公式

令：

```text
p = clamp(priority / 100, 0, 1)
```

当可估算剩余量时：

```text
remainingTerm = -lengthSlope * log2(remainingUnits / neutralUnits)
priorityTerm  = priorityCadenceWeight * (0.5 - p)

targetAF = clamp(
  minAF,
  maxAF,
  baseAF(cardType) + remainingTerm + priorityTerm + explicitPaceTerm
)

newAF = (1 - learningRate) * oldAF + learningRate * targetAF
```

符号含义：

- 剩余内容越多，`remainingTerm` 越负，AF 越小，材料保持更频繁；
- 越接近完成，AF 越大，材料更快淡出；
- priority 越高，`priorityTerm` 越负，间隔增长更慢；
- priority 只弱影响 cadence，主要仍影响当天排序。

Shadow 阶段可使用的参数种子，仅用于比较，不是生产默认：

| 参数 | Topic | Extract | 说明 |
| --- | ---: | ---: | --- |
| `baseAF` | 1.45 | 1.65 | 短 Extract 更快拉开 |
| `minAF` | 1.05 | 1.05 | 防止不增长或倒退 |
| `maxAF` | 2.20 | 2.20 | 防止一次跳得过远 |
| `lengthSlope` | 0.12 | 0.08 | Extract 对长度较不敏感 |
| `priorityCadenceWeight` | 0.30 | 0.30 | 最大约 ±0.15 |
| `learningRate` | 0.25 | 0.25 | 避免 AF 剧烈变化 |

这些数值必须经 shadow replay 与真实日志校准。

### 11.3 剩余量估计

优先级顺序：

1. Book IR 已知章节页码、章节结束位置或明确进度；
2. Topic 的 `resumeBlockId / breakpoint` 在有界 descendants 中的位置；
3. Extract 的当前内容长度与精炼状态；
4. 无可靠数据时，不使用 `remainingTerm`，回退 type base AF。

禁止为了计算 remaining 对全仓库发起无界子块读取。

### 11.4 下一间隔

真实进展时：

```text
nextInterval = clamp(
  typeMinInterval,
  typeMaxInterval,
  round(currentIntentionalInterval * newAF)
)
```

`currentIntentionalInterval` 不包含：

- 首次 sibling queue delay；
- 手动 postpone 产生的临时 delay；
- auto-postpone 产生的临时 delay。

第一阶段仍保留当前 Topic 60 天 / Extract 30 天上限，先观察 shadow 数据。是否提高或改为用户配置，必须依据真实 revisit / postpone 数据决定，不能只因 SuperMemo 支持数千天就直接放大。

### 11.5 Stall guard

普通 Topic 与 Extract 应获得和顺序 Book IR 一致的“无进展不视为成功”语义：

```text
if action == next && progressAdvanced == false:
  nextInterval = currentIntentionalInterval
  AF 不上调
```

连续无进展可以：

- 保持 interval；
- 提示用户选择 Soon / Later / 降低 priority / 归档；
- 进入漏斗诊断；
- 不得静默模拟一次成功处理。

## 12. 动作语义

| 动作 | AF | `intervalDays` | `due` | 其他 |
| --- | --- | --- | --- | --- |
| 下一篇且有进展 | 平滑到 target AF | 乘新 AF | interval + due jitter | 增加 readCount，推进 stage |
| 下一篇但无进展 | 不上调 | 保持 | 按保持后的 interval | 记录 stall |
| 创建 Extract | 父卡可记录真实进展；不因数量直接强调 AF | 父卡按正常 next 语义 | 子 Extract 首次错峰 | 子卡继承 priority 时加阻尼 |
| Extract 转 Item | 不再使用 | 清理 IR 调度 | 由 FSRS 接管 | 保留来源追溯 |
| 调整 priority | 下一次动作时影响 target AF | 默认不立即大幅重写 | 可提供显式“同时重排”选项 | priority 立即影响排序 |
| 手动 Postpone | 不更新 | 不修改 intentional interval | 只移动 due | 增加 postponeCount，结果可见 |
| Auto-postpone | 不更新 | 不修改 intentional interval | 只移动旧积压 due | 批次、成功/失败、可撤销 |
| 今天阅读 | 不更新 | 不修改 | due 改为今天 | 不伪造 readCount |
| Soon / Normal / Later | `explicitPaceTerm` 为负 / 0 / 正 | 下次真实动作采用 | 正常生成 | 可选二期交互 |

### 12.1 Extract priority 继承

SuperMemo 公开资料只确认 Extract priority 受父文章影响，没有公开精确映射。

候选策略：

```text
childPriority = round(
  inheritWeight * parentPriority
  + (1 - inheritWeight) * defaultPriority
)
```

建议 shadow seed：`inheritWeight=0.8`。

目的：

- 保留来源价值；
- 避免高优 Topic 一次产生十几个同优先级 Extract；
- 用户仍可即时手动提权。

这项需要与现有“Extract 完全继承 Topic priority”的用户预期一起 Review。

## 13. 今日队列评分

### 13.1 候选资格

第一层只回答“今天是否可选”：

```text
eligible = active && due <= localDayEndOrPolicyBoundary
```

必须沿用当前本地自然日语义，并明确今天到期与旧积压的分类。

### 13.2 候选评分

候选公式：

```text
daysOverdue = max(0, localToday - dueDay)

overdueBonus = min(20, 5 * ln(1 + daysOverdue))
starvationBonus = min(10, 2 * sqrt(eligibleSkipCount))

score = priority
      + overdueBonus
      + starvationBonus
      + boundedExplorationNoise
```

设计理由：

- priority 的 100 分尺度仍是主导；
- 逾期越久加分越多，但上限 20，低价值旧材料不能无限压过高价值材料；
- 长期有资格却未入选的材料最多获得 10 分补偿；
- 随机只在有限范围内改变相近候选。

`eligibleSkipCount` 在没有可靠日志前设为 0，不得用手动 postponeCount 冒充。

### 13.3 保护区与随机

- `priority >= protectedThreshold` 的候选不加随机噪声；
- 其他候选只允许 `[-3, +3]` 的稳定噪声；
- seed 使用 `blockId + localDate + frozenSessionDescriptor`；
- 同一会话重复计算顺序不变；
- 新会话或跨天可以产生小幅变化；
- 不使用 UTC 日期；
- 不允许随机把低优材料直接越过保护区。

Logseq 的 Beta priority 可作为远期实验，但第一版使用有界噪声更易解释和测试。

### 13.4 约束优先级

建议明确为：

```text
1. 数据有效与 eligibility
2. 用户 focus（必入；若成本超过预算，则本次只放 focus）
3. 当日剩余额度与冻结 scope
4. 时间预算
5. 高优先级保护
6. Source Topic 最低曝光
7. 新 Extract 最终最大比例
8. 普通 score 填充
9. 至少一次有界探索（队列规模达到阈值时）
```

所有比例约束必须在最终队列上复核。约束冲突时返回诊断，例如：

```text
topicFloorSatisfied: false
reason: "time_budget_exhausted_by_protected_items"
```

### 13.5 时间预算

保留当前成本估计：

- 卡种基线；
- 停留历史 EWMA；
- 最少样本门槛；
- focus 成本也计入预算；
- 探索替换后重新计算总成本；
- 最终误差不得超过一个最小可选条目的估计成本。

不建议直接按 `score / cost` 全局排序，因为这会系统性饿死长文章。成本用于装箱与约束，不应替代用户价值。

## 14. Overload / Auto-postpone

### 14.1 执行时机

```text
收集候选
  -> 冻结 focus / scope / budget / seed
  -> 生成最终会话队列
  -> 计算未选中的旧积压
  -> 若设置启用且今日未执行过，再执行 auto-postpone
```

### 14.2 可推迟集合

只能包含：

- `dueDay < localToday` 的旧积压；
- 未进入冻结会话；
- 非 focus；
- 非保护区；
- 未超过 postpone count / interval 等保护条件；
- 当前 scope 内。

今天刚到期的内容不自动推迟。

### 14.3 推迟方式

第一版复用现有按 cardType / priority 产生 delay window 的逻辑，但改变写入语义：

- 只移动 `due`；
- 不把 delay 写入 `intervalDays`；
- 不更新 AF；
- 不增加 readCount；
- 写入 `lastAction=autoPostpone` 与批次 ID；
- 每张写入后失效 cache；
- 返回实际成功、失败和回滚结果。

### 14.4 每日一次

同一 local day、同一 repo、同一策略版本最多自动处理一次。重新打开面板不能反复把积压越推越远。

每日标记应持久化或能从可靠日志推导，不能只放进程内 Map 后假设插件不会重载。

## 15. 每日额度、会话与混合 SRS

### 15.1 真实 IR 每日额度

`dailyLimit` 应表示本地当天最多完成的唯一 IR 条目数，而不是每次生成上限。

要求：

- 从已确认的 IR 动作日志扣减；
- 同一卡当天多次打开只消费一次；
- 失败动作不消费；
- 刷新、重开、换面板不重置；
- `0` 仍表示不限；
- repo 隔离。

### 15.2 冻结 IR session descriptor

至少包含：

- repo；
- local date；
- IR-only / mixed；
- scope；
- focus；
- time budget；
- count budget；
- Source Topic floor；
- new Extract cap；
- exploration；
- mixed SRS ratio；
- seed / policy version。

若未来支持动态补入，新条目仍必须遵守这份冻结 descriptor 和剩余 daily budget。

### 15.3 混合 SRS

混合模式必须复用普通 SRS 的：

- frozen scope；
- 今日 new / review remaining budget；
- due 排序；
- card identity；
- 已处理日志去重。

不得重新实现一套弱化版 SRS budget。

三个比例必须分开命名和解释：

1. Source Topic 在纯 IR 中的最低比例；
2. 新 Extract 在纯 IR 中的最大比例；
3. 普通 SRS Items 在 mixed session 中的比例。

SuperMemo 的约 1 Topic : 4 Items 只作为混合模式参考，不应强加给专门的 IR-only 时间盒。

## 16. Shadow Mode 与可解释日志

### 16.1 Shadow Mode

在不改变真实顺序的情况下，同时计算：

- 当前 v1 入选队列；
- v2 候选队列；
- 每张卡的 score terms；
- v1 / v2 分歧位置；
- v2 的预测时间成本；
- v2 建议 AF 与 interval，但不写 block properties。

### 16.2 日志建议

本地调度日志最小形状：

```ts
type IRSelectionTrace = {
  repo: string;
  localDate: string;
  sessionId: string;
  cardId: string;
  cardType: "topic" | "extracts";
  due: number;
  priority: number;
  daysOverdue: number;
  aFactor?: number;
  estimatedCostSeconds: number;
  scoreTerms: {
    priority: number;
    overdue: number;
    starvation: number;
    exploration: number;
  };
  selected: boolean;
  reason: string;
};
```

这是设计形状，不代表必须新建同名文件或直接持久化全部字段。实现时应复用现有日志基础设施，并保持 enqueue / flush 失败可见。

### 16.3 评估指标

至少记录：

- 时间预算预测值与实际值；
- 单位分钟的阅读进展；
- 每次 Topic 产生的 Extract 数；
- Extract 到 Item 的中位 / P90 时间；
- 下一篇但无进展的比例；
- 手动 Soon / Later / Postpone / priority 调整率；
- 各 priority 分桶的 P50 / P95 overdue days；
- 长期有资格但未入选的条目数；
- auto-postpone 实际成功、失败、回滚数；
- Topic / Extract / SRS 的实际会话占比；
- 同源 Extract 在同一天重复出现的比例。

不把停留时长直接当价值，只用于成本估算。

## 17. 分阶段实施计划

### P0：修复语义和可测性

范围：

- `enableAutoDefer` 真正门控自动写入；
- focus 在 overload 前冻结；
- 主时间盒读取正确配额设置；
- 修复新 Extract 最终 cap；
- 统一 priority interval helper 与 clamp；
- sibling delay 只影响首次 due；
- 修复探索成本；
- local date seed；
- 混合 SRS 复用 budget；
- 真实 daily remaining；
- 失败结果按实际写入计数；
- 创建路径 cache invalidation。

退出条件：

- 相关单元和 hook 集成测试通过；
- 关闭 auto-defer 时打开会话无任何 due 写入；
- focus 永不被同批次 postpone；
- 重开会话不会重置 daily remaining。

### P1：Selection Trace + Shadow Queue

范围：

- 冻结 IR session descriptor；
- 记录 v1 选择原因；
- 计算 v2 score，但不改变顺序；
- 记录时间成本和真实动作。

退出条件：

- 连续使用 2 至 4 周；
- 日志 flush 失败可见且可重试；
- 可以离线 replay 相同 session 得到相同 v2 队列。

### P2：动态 A-Factor

范围：

- `ir.aFactor` lazy migration；
- progress-aware target AF；
- stall guard；
- postpone 与 intentional interval 分离；
- 继续保留旧 interval caps。

退出条件：

- 迁移不批量改 due；
- feature flag 关闭时与 v1 输出一致；
- 旧数据缺字段时不报假成功或静默丢卡；
- 真实 Orca 属性写后 cache 复读一致。

### P3：启用 v2 Selection

范围：

- bounded overdue aging；
- protected zone；
- 有界每日随机；
- starvation bonus（有可靠日志后）；
- 最终配额验证；
- 选择原因 UI / 诊断。

退出条件：

- 同 session 可复现；
- 保护区不被随机越过；
- 低 priority 的极端 overdue 不会无限压过高 priority；
- 短队列也能执行最小探索；
- 时间预算保持硬约束。

### P4：Overload 与 Daily Budget

范围：

- 每日一次 auto-postpone；
- 只处理此前日期旧积压；
- due-only delay；
- 持久化批次与跨重载撤销策略；
- 真实 IR daily budget；
- mixed SRS budget 对齐。

### P5：参数校准

依据 shadow / v2 日志调整：

- `baseAF`；
- `minAF / maxAF`；
- priority cadence weight；
- overdue cap；
- exploration range；
- Topic / Extract interval caps；
- Source Topic floor；
- new Extract cap；
- mixed SRS ratio。

禁止仅凭主观感觉一次改多个参数。每轮只改一类变量并保留 replay 对照。

## 18. 迁移与回滚

### 18.1 迁移原则

- 不批量扫描并重写整个仓库；
- 不改变现有 due；
- 不改变现有 interval；
- 缺 `ir.aFactor` 时按卡种 lazy 初始化；
- 新字段写入必须 cache invalidate；
- 迁移失败保持旧数据可继续由 v1 读取；
- 不清理未知字段或用户属性。

### 18.2 Feature Flag

建议独立控制：

- `shadow`：只计算、不改变；
- `adaptive_interval`：启用动态 AF；
- `adaptive_queue`：启用 v2 selection；
- `auto_postpone`：启用每日一次旧积压治理。

这些是能力概念，最终 setting key 必须经 schema review，不能直接照抄为新增属性。

### 18.3 回滚

关闭 v2 时：

- Topic 继续 `×1.25`；
- Extract 继续 `×1.35`；
- 忽略但不删除 `ir.aFactor`；
- 保留已产生的合法 due / interval；
- 不执行反向批量迁移；
- Selection 恢复 v1，但 daily budget 和 P0 correctness 修复不回滚。

## 19. 测试计划

### 19.1 纯函数测试

- AF 初始化：Topic `1.25`、Extract `1.35`；
- 高 priority 得到较低 target AF；
- remaining 越多，AF 越低；
- AF clamp 与 EWMA；
- stall 不增长 interval；
- postpone 不修改 intentional interval / AF；

- 最终新 Extract cap 不能被 fill 阶段绕过；
- Topic floor 冲突返回诊断；

### 19.2 Hook / 会话集成测试

- auto-defer off 无写入；
- auto-defer on 只处理旧积压；

- session descriptor 在刷新 / 补队列时保持；
- daily remaining 跨会话扣减；
- mixed SRS 不绕过普通 daily budget；
- 当前日 due 不被 auto-postpone；

### 19.3 持久化与失败测试

正式修改属性、索引、缓存或队列前，实施 agent 必须：

1. 阅读 `plugin-docs/modules.md` 与相关 backend / property 文档；
2. 提供一次性、只读、top-level `await` 的 Console 脚本；

- 一个普通 Topic；

- 一个 priority 90 与一个 priority 20 的条目；

- 一张已转 Cloze 的原 Extract；
- auto-defer 开 / 关各一次；
- 同一天关闭重开会话一次。

## 21. 验收标准

### 21.1 Correctness

- auto-defer off 时，打开工作区和创建时间盒不会改任何 due；
- focus 卡不会被同批 overload 写入；
- 新 Extract 最终占比不超过配置；
- Source Topic floor 能满足时必须满足，不能满足时有原因；
- dailyLimit 跨会话不重置；
- mixed SRS 不超过普通 SRS remaining budget；
- today due 不被 auto-postpone；
- sibling delay 只影响首次 due；
- 所有失败数来自真实写入结果；
- 所有每日 seed 使用本地日。

### 21.2 Algorithm

- 高 priority 在相同 due / cost 下稳定靠前；
- overdue boost 有界；
- protected zone 不受探索影响；
- 无进展不会增长 interval；
- 长 Topic 的 AF 通常低于短 Extract，但 priority 可以弱纠偏；
- v2 关闭时保留 v1 行为；
- shadow replay 可复现。

### 21.3 Product

- 用户能知道“为什么今天看到这张”；
- 用户能区分“阅读间隔”和“临时推后”；
- 时间盒不因 hover、加载文字或动态内容改变布局或预算；
- 错误不会显示成“暂无到期内容”或“全部推后成功”；
- 自动行为可关闭，关闭后立即生效；
- 不要求用户理解 AF 数学细节才能使用。

### 21.4 Metrics

算法是否更好不能只看“队列更整齐”。至少比较：

- 单位时间有效进展；
- 无进展率；
- Extract 转 Item 延迟；
- 手动 Postpone / priority 调整率；
- 高 priority overdue P95；
- 长期饥饿条目；
- 时间预算偏差；
- 同源重复密度；
- overload 写入失败率。

在没有 baseline 前不预设虚假的提升百分比。

## 26. 最终建议

推荐实现顺序不可颠倒：

```text
P0 修语义与预算
  -> P1 加冻结 descriptor、trace 与 shadow queue
  -> P2 动态 A-Factor
  -> P3 启用有界 aging / exploration
  -> P4 每日一次旧积压治理
  -> P5 用真实数据调参
```

第一轮实现应只做 P0 + P1。没有可 replay 的 shadow 数据前，不建议同时上线动态 AF、随机探索和新的 auto-postpone；否则出现体验变化时无法判断是哪一层造成的。
