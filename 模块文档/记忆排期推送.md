## 0. 这份文档解决什么问题？

在渐进阅读（IR）里，用户经常会在同一次阅读中做出很多摘录（Extract）。如果这些摘录的第一次复习时间集中在 1～2 天内，会产生明显的心理压力（“那天会爆炸”）。

本项目目前的目标是：

- 当用户在**短时间内**为**同一个 Topic** 一口气做出很多 Extract（例如 10～20 个）时，第一次复习时间应**随机铺开**到更长窗口（例如 7 天，必要时可到 10～14 天后）。
- 用户可接受“晚一点也行”（阅读材料不必精确卡在遗忘临界点）。
- 只要别挤在 1～2 天里，不追求严格均匀分布。

## 1. 两个关键概念：排期 vs 推送

### 1.1 排期（Scheduling）

“排期”指系统把下一次到期时间写入块属性：

- `ir.intervalDays`：下一次间隔（天，允许小数）
- `ir.due`：下一次到期时间（Date）

排期属于“写入行为”，主要入口在：

- `src/srs/incrementalReadingStorage.ts`（兼容门面）
  - 实现：`irStatePersistence.ts` / `irSchedulingMutations.ts` / `irSchedulingHelpers.ts`
  - `ensureIRState`
  - `updatePriority`（新卡 init/migrate 时应用分散 + 排队延迟）
  - `markAsRead` / `markAsReadWithPriority`
  - `postpone`

基础间隔与推迟天数：

- `src/srs/incrementalReadingScheduler.ts`
  - `getBaseIntervalDays` / `getPostponeDays` / `calculateNextDue`（后者含 0–12h 随机 jitter，用于部分路径）

分散：

- `src/srs/incrementalReadingDispersal.ts#computeDispersedIntervalDays`

### 1.2 推送（Queue Push）

“推送”指系统在“今天”把哪些卡片放入阅读面板队列（**不一定**会写回排期）。

- 入口：`src/srs/incrementalReadingCollector.ts`
  - 收集到期卡：`collectIRCardsFromBlocks` 等
  - 构建今日队列：`buildIRQueue`
  - “溢出推后”（按钮触发，写回排期）：`deferIROverflow` / 相关 defer API
- 策略纯函数：`src/srs/incremental-reading/irQueuePolicy.ts`（时间预算、Topic 最低曝光、新 Extract 配额、稳定随机）
- 混合会话：`src/srs/incremental-reading/irMixedQueuePolicy.ts`（阅读条目 + 到期复习卡）

> 重点：打开面板/构建队列通常只是“展示”，不会自动修改 `ir.due`；真正写回发生在“用户动作”（已读/推后/调优先级）或“明确按钮”（溢出推后）。

## 2. SuperMemo 概念对照（我们借鉴什么，不等同复刻）

下述内容用于“理解方向”，**不是**逐行复刻 SuperMemo 的私有实现：

### 2.1 Random Dispersal（随机分散）

要点：允许间隔围绕“理论最优值”波动，从而减少同日堆积（lumpiness）。

对我们而言：Extract 属于阅读材料（Topic-like），时间精确性要求较低，可以接受更大的分散窗口。  
**已落地**：`computeDispersedIntervalDays`。

### 2.2 Spread / Branch（分支分散）

要点：当你从同一篇材料（同一 Topic）提取出一批子内容，这批内容天然是同一分支（siblings）。

对我们而言：同一 Topic 下同一天产生的多个 Extract，应该有“排队延迟”，第 2 个、第 3 个……自然更晚出现。  
**已落地**：`computeNewExtractQueueDelayDays`（仅新 Extract 初始化阶段）。

### 2.3 Auto-Postpone（自动推迟）

要点：如果某天任务过载，保留高价值/高优先级的，低优先级自动推迟到未来。

对我们而言：削峰填谷的兜底——**已有**“溢出推后”按钮能力；**完整自动全库 postpone** 仍属增强方向。用户偏好是 **A：创建时就铺开**，因此“分支分散”放在创建/初始化阶段完成大头。

## 3. 我们的优先级语义（避免与 SM 混淆）

本项目：

- `ir.priority`：0～100，**数值越大越优先**（越频繁、越靠前）
- 唯一真相在 `ir.priority` 属性；不从 `#card` 的 ref.data 读优先级（见 scheduler 文件头注释）
- 这与很多系统把“数字越小越重要”的习惯相反

## 4. 当前实现：从“基础间隔”到“最终到期”

### 4.1 基础间隔（Base Interval）

基础间隔由 `cardType + ir.priority` 线性插值后取整（`getBaseIntervalDays`）：

| cardType | priority 0 | priority 100 |
|----------|------------|--------------|
| Topic | 约 14 天 | 约 2 天 |
| Extract（`extracts`） | 约 7 天 | 约 1 天 |

举例（优先级 50）：Extract 基础间隔约为 `4` 天（`round(lerp(7,1,0.5))`）。

### 4.2 分散抖动（Dispersal）

实现：`computeDispersedIntervalDays`

核心做法：

1. 使用“按天稳定”的随机 seed（`blockId + localDayStart + cardType + salt`，FNV-1a + mulberry32），同一天重复触发不会反复变化。
2. 新卡（`isNew`）：**只向后**分散  
   - Topic：最多约 1 天  
   - Extract：最多 `base * 0.5`  
   - 再加可选 `queueDelayDays`
3. 非新卡：按比例 ± 分散（Extract ratio 0.35 maxAbs 3d；Topic ratio 0.2 maxAbs 2d）

### 4.3 新 Extract 的“分支排队延迟”

实现：`irSchedulingHelpers.ts` 内 `computeNewExtractQueueDelayDays`（仅供内部排期模块使用，不经 `incrementalReadingStorage` 门面导出）；在**新 Extract** 且 `lastAction` 为 `init`/`migrate` 的 `updatePriority` 路径注入。

规则：

- 父块须为 Topic；计算前 **强制失效父块缓存** 再读 `children`（避免快速连续创建时全员 `index=0`）
- 统计同一自然日、创建时间更早（或同毫秒但 id 更小）的兄弟块个数 → `index`
- `stepDays = min(0.5, max(0.15, baseIntervalDays * 0.2))`
- `queueDelayDays = index * stepDays`

直觉：

- 10 个 Extract：最后一个大约比第一个多推迟数天（step 接近 0.5 时约 4.5 天）
- 20 个 Extract：可能再翻倍量级

### 4.4 最终间隔与到期时间

写回时 clamp：

- Extract 最大 **30** 天（`EXTRACT_MAX_INTERVAL_DAYS`）
- Topic 最大 **60** 天（`TOPIC_MAX_INTERVAL_DAYS`）
- 最小 1 天
- `ir.due = baseDate + intervalDays`（`computeDueFromIntervalDays`）

非新卡调优先级：按旧/新 priority 比例修正已有 interval（factor 限制 0.6..1.6），**不再**叠加排队延迟。

## 5. “你做了 10 个 Extract”时的预期结果（验收直觉）

假设：

- 今天是 `2/2`（举例）
- 这批 Extract 都继承到 `ir.priority=50`
- Extract 基础间隔 `base≈4` 天

那么第 N 个（N 从 1 到 10）Extract 的第一次间隔大致为：

- `base (4d) + 随机往后(0~2d) + 排队延迟((N-1)*stepDays)`

得到区间量级：

- 第 1 个：约 `4.0~6.0` 天后
- 第 5 个：约 `6.0~8.0` 天后
- 第 10 个：约 `8.5~10.5` 天后

若全部挤在 `4~6` 天附近，优先查排队延迟是否为 0（§6）。

## 6. 调试清单：为什么还会堆积？

### 6.1 最常见：排队延迟算出来是 0

可能原因：

- 父 Topic 的 `children` 列表拿到的是旧快照（实现已对 parent 做 cache invalidate；若仍失败查创建顺序/父类型）
- 新创建的 Extract 块拿不到 `created` 时间
- 创建流程没走到会应用 queue delay 的 `updatePriority`（`createExtract` / `autoMarkAsExtract` 等应触发 ensure/priority 路径）

快速定位：

1. UI 上看每个 Extract 的间隔是否随创建序号上升
2. 临时日志打印：`blockId` / `parentId` / `created` / `index` / `stepDays` / `queueDelayDays`
3. 快速创建时父 `children.length` 是否增长

### 6.2 “乱序”是正常的

随机铺开，不要求严格均匀；不挤在 1～2 天窗口即达标。

### 6.3 每日上限 / 时间预算不会自动改排期

`buildIRQueue` + `irQueuePolicy` 负责“今天展示谁”，**不会**自动重排所有卡的 `ir.due`。

需要“当天太多就推迟低优先级”时：用**溢出推后**按钮；全自动 Auto-Postpone 仍为增强方向。

### 6.4 混合会话复习侧

`irMixedQueuePolicy` 混入的是**已到期非新** SRS 复习卡（精确 `due.getTime() <= now`），与 IR 阅读条目交织；不改变 IR 排期写入规则。
