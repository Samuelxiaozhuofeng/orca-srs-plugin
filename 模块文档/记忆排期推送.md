## 0. 这份文档解决什么问题？

在渐进阅读（IR）里，用户经常会在同一次阅读中做出很多摘录（Extract）。如果这些摘录的第一次复习时间集中在 1～2 天内，会产生明显的心理压力（“那天会爆炸”）。

本项目目前的目标是：

- 当用户在**短时间内**为**同一个 Topic** 一口气做出很多 Extract（例如 10～20 个）时，第一次复习时间应**随机铺开**到更长窗口（例如 7 天，必要时可到 10～14 天后）。
- 用户可接受“晚一点也行”（阅读材料不必精确卡在遗忘临界点）。
- 只要别挤在 1～2 天里，不追求严格均匀分布。

## 1. 两个关键概念：排期 vs 推送

### 1.1 排期（Scheduling）

“排期”指系统把下一次到期时间写入块属性：

- `ir.intervalDays`：下一次间隔（天，允许小数）
- `ir.due`：下一次到期时间（Date）

排期属于“写入行为”，主要入口在：

- `src/srs/incrementalReadingStorage.ts`（兼容门面）
  - 实现：`irStatePersistence.ts` / `irSchedulingMutations.ts` / `irSchedulingHelpers.ts`
  - `ensureIRState`
  - `updatePriority`（新卡 init/migrate 时应用分散 + 排队延迟）
  - `markAsRead` / `markAsReadWithPriority`
  - `postpone`

基础间隔与推迟天数：

- `src/srs/incrementalReadingScheduler.ts`
  - `getBaseIntervalDays` / `getPostponeDays` / `calculateNextDue`（后者含 0–12h 随机 jitter，用于部分路径）

分散：

- `src/srs/incrementalReadingDispersal.ts#computeDispersedIntervalDays`

### 1.2 推送（Queue Push）

“推送”指系统在“今天”把哪些卡片放入阅读面板队列（**不一定**会写回排期）。

- 入口：`src/srs/incrementalReadingCollector.ts`
  - 收集到期卡：`collectIRCardsFromBlocks` 等
  - 构建今日队列：`buildIRQueue`（旧日限混排；仍读 `topicQuotaPercent` 作目标占比）
  - “溢出推后”（按钮触发，写回排期）：`deferIROverflow` / 相关 defer API
- 时间盒路径：`useIRWorkspaceSession` → `selectQueueWithPolicy`（`irQueuePolicy.ts`）→ `assembleSessionReadingQueue`（focus + `dailyLimit`）
  - `topicQuotaPercent`（0..100 clamp）→ `topicMinRatio`（Source Topic 在纯 IR 阅读队列的最低曝光）
  - 比例类约束在**最终队列**上成立；冲突时 `SelectQueueResult.diagnostics` 给出机器可测 reason
  - seed：`formatLocalDateKey(sessionStartedAt)` 本地日
  - **Batch B1**：创建/打开/刷新/重试会话**只读**；不调用 `applyAutoPostpone`；主收集与 focus 回退均 `{ readOnly: true }`（跳过 `ensureIRState`，仅 `loadIRState` 内存解析；**非**持久化完成）；`enableAutoDefer` 不门控会话写入（它只控制资料库按钮）
- 混合会话：`src/srs/incremental-reading/irMixedQueuePolicy.ts`（阅读条目 + 到期复习卡）

> 重点：打开面板/构建时间盒队列是“展示与装配”，**不会**因会话启动自动修改 `ir.due` 或其它 block property；真正写回发生在“用户动作”（已读/推后/调优先级）或资料库“明确按钮”（溢出推后 / `deferIROverflow`），以及**非只读**收集路径上的惰性 ensure。运行时只读证据曾确认部分 Topic 在 `orca.state.blocks` 未命中但 `get-block` 可读完整 `ir.*`——会话路径须容忍 state cache miss；**不**据此宣称 Orca 实例已验证 B1 修复。

## 2. SuperMemo 概念对照（我们借鉴什么，不等同复刻）

下述内容用于“理解方向”，**不是**逐行复刻 SuperMemo 的私有实现：

### 2.1 Random Dispersal（随机分散）

要点：允许间隔围绕“理论最优值”波动，从而减少同日堆积（lumpiness）。

对我们而言：Extract 属于阅读材料（Topic-like），时间精确性要求较低，可以接受更大的分散窗口。
**已落地**：`computeDispersedIntervalDays`。

### 2.2 Spread / Branch（分支分散）

要点：当你从同一篇材料（同一 Topic）提取出一批子内容，这批内容天然是同一分支（siblings）。

对我们而言：同一 Topic 下同一天产生的多个 Extract，应该有“排队延迟”，第 2 个、第 3 个……自然更晚出现。
**已落地**：`computeNewExtractQueueDelayDays`（仅新 Extract 初始化阶段）。

### 2.3 Auto-Postpone（自动推迟）

要点：如果某天任务过载，保留高价值/高优先级的，低优先级自动推迟到未来。

对我们而言：削峰填谷的兜底——**已有**“溢出推后”按钮能力；**完整自动全库 postpone** 仍属增强方向。用户偏好是 **A：创建时就铺开**，因此“分支分散”放在创建/初始化阶段完成大头。

## 3. 我们的优先级语义（避免与 SM 混淆）

本项目：

- `ir.priority`：0～100，**数值越大越优先**（越频繁、越靠前）
- 唯一真相在 `ir.priority` 属性；不从 `#card` 的 ref.data 读优先级（见 scheduler 文件头注释）
- 这与很多系统把“数字越小越重要”的习惯相反

## 4. 当前实现：从“基础间隔”到“最终到期”

### 4.1 基础间隔（Base Interval）

基础间隔由 `cardType + ir.priority` 线性插值后取整（`getBaseIntervalDays`）：

| cardType | priority 0 | priority 100 |
|----------|------------|--------------|
| Topic | 约 14 天 | 约 2 天 |
| Extract（`extracts`） | 约 7 天 | 约 1 天 |

举例（优先级 50）：Extract 基础间隔约为 `4` 天（`round(lerp(7,1,0.5))`）。

### 4.2 分散抖动（Dispersal）

实现：`computeDispersedIntervalDays`（**只**返回 intentional cadence）

核心做法：

1. 使用“按天稳定”的随机 seed（`blockId + localDayStart + cardType + salt`，FNV-1a + mulberry32），同一天重复触发不会反复变化。
2. 新卡（`isNew`）：**只向后**抖动 base
   - Topic：最多约 1 天
   - Extract：最多 `base * 0.5`
3. 非新卡：按比例 ± 分散（Extract ratio 0.35 maxAbs 3d；Topic ratio 0.2 maxAbs 2d）
4. **不再**把 `queueDelayDays` 加进返回的 interval（即使参数仍被传入也会被忽略）

组合入口：`computeDispersedSchedule` → `{ intervalDays, due, queueDelayDays }`，其中
`due = baseDate + intervalDays + queueDelayDays`。

### 4.3 新 Extract 的“分支排队延迟”（due-only）

实现：`irSchedulingHelpers.ts` 内 `computeNewExtractQueueDelayDays` / `resolveExtractSourceTopicId` / `countEarlierSameDayExtractSiblings`；在**新 Extract** 且 `lastAction` 为 `init`/`migrate` 的 `updatePriority` 路径注入。

规则（Batch B2）：

- **来源 Topic**：优先读 `ir.sourceTopicId`；仅在属性缺失时回退「直接 parent 为 Topic」。真实 Orca 证据：Extract 可嵌套在正文块下（Topic `#231` → 正文 `#237` → Extract `#1646`），旧“parent 即 Topic”假设会返回 0 delay。
- 有界 BFS：`maxDepth/maxBlocks/concurrency` **硬 cap**（NaN/Infinity 安全钳制到默认上限）；初始 queue 与扩展均 ≤ maxBlocks
- **同源过滤**：有 `ir.sourceTopicId` 须等于当前 source；缺失时仅计 direct parent 即 source 的 legacy Extract；遇到嵌套 **另一 Topic** 不深入其子树
- **截断可见**：触 cap 时 `console.warn`（sourceTopicId / maxDepth / maxBlocks），不静默伪装完整扫描
- 统计同一自然日、更早创建的同源 Extract → `index`
- `stepDays = min(0.5, max(0.15, baseIntervalDays * 0.2))`
- `queueDelayDays = index * stepDays` → **只加在首次 due**，**不**写入 `ir.intervalDays`

创建顺序（`initializeExtractScheduleAfterCreate`）：

1. `invalidateIrBlockCache`（刚写的 #card/type=extracts 对 ensure 可见）
2. 写 `ir.sourceTopicId`（若有）→ 再 invalidate
3. `ensureIRState`
4. `updatePriority`（此时才算 sibling delay）

直觉：

- 10 个 Extract：最后一个首次 due 大约比第一个多推迟数天（step 接近 0.5 时约 4.5 天）
- 后续「下一篇」增长只从 intentional interval 起乘 1.35

### 4.4 最终间隔、到期与推后

写回时 clamp（intentional interval）：

- Extract 最大 **30** 天（`EXTRACT_MAX_INTERVAL_DAYS`）
- Topic 最大 **60** 天（`TOPIC_MAX_INTERVAL_DAYS`）
- 最小 1 天
- 新 Extract 首次：`ir.due = baseDate + intervalDays + queueDelayDays`；`ir.intervalDays` **不含** queueDelay

非新卡调优先级：`adjustIntervalForPriorityChange(cardType, …)`（factor 0.6..1.6 + 卡种 clamp）；`performPriorityAdjust` 与 `updatePriority` 同一实现。

**推后只移 due**（手动 / 自动 / 资料库溢出）：

- 保留 `intervalDays` 与 `position`（overflow **不得**重排 Topic position）
- 更新 `due`、`postponeCount`、`lastAction`（及自动批次 `autoPostponeBatchId`）
- 不增加 `readCount`；不借推后重置 intentional interval
- 自动推后中途失败抛 `AutoPostponeError`；若 snapshot **回滚也失败**，`rollbackFailed` 列出未恢复 blockId——**不**伪装已恢复，后续需可见处理

### 4.5 两类节奏：记忆型 Topic vs 顺序激活章 SAC

| 类型 | 何时 | 基线 | 「下一篇」后 |
|------|------|------|-------------|
| **普通 Topic / Extract / distributed 章节** | 默认 | `getBaseIntervalDays`（Topic p50≈8 天） | Topic ×1.25（上限 60）；Extract ×1.35（上限 30） |
| **顺序 Book IR 当前 `activeChapterId` 章（SAC）** | `mode=sequential` 且块为激活章 | `1 + 2*(1 - priority/100)` → p0/50/100 ≈ **3/2/1 天** | **不** ×1.25；按 priority + 停滞计数重算；上限约 **6 天** |

要点：

- **目标不同**：Topic 增长偏向「越读越不急着再来」的记忆间隔；SAC 偏向「继续推进当前章」的短节奏。1/2/3 天是启发式，不是科学最优值。
- **停滞保护**：连续 next 但 `resumeBlockId`/`readingBreakpoint` 指纹不变 → 逐渐加长间隔至约 6 天，降低空转章对**全局队列预算**的占用；与队列挑选（collector / queue policy）独立——SAC 只改 due 写入，不改预算公式。
- **手动推后**：`postpone` 写 due 后，打开面板/加载状态**不会**用 SAC 覆盖；下次「下一篇」才重算。
- **兼容**：不重写旧 plan；旧 sequential 章从下一次用户写路径启用 SAC，不批量 re-anchor 远期 due。
- **`totalDays`**：仍只服务 distributed 首次分散；sequential 存盘但不参与调度（见 Book IR 文档）。

## 5. “你做了 10 个 Extract”时的预期结果（验收直觉）

假设：

- 今天是 `2/2`（举例）
- 这批 Extract 都继承到 `ir.priority=50`
- Extract 基础间隔 `base≈4` 天

那么第 N 个（N 从 1 到 10）Extract：

- **`ir.intervalDays`** ≈ `base (4d) + 随机往后(0~2d)`（**不含**排队延迟）
- **首次 due 偏移** ≈ `intervalDays + (N-1)*stepDays`

得到首次 due 区间量级：

- 第 1 个：约 `4.0~6.0` 天后
- 第 5 个：约 `6.0~8.0` 天后
- 第 10 个：约 `8.5~10.5` 天后

若全部挤在 `4~6` 天附近，优先查排队延迟是否为 0（§6），以及 `ir.sourceTopicId` 是否已在 `updatePriority` 前写入。

## 6. 调试清单：为什么还会堆积？

### 6.1 最常见：排队延迟算出来是 0

可能原因：

- **嵌套 Extract**：parent 是正文块而非 Topic，且 `ir.sourceTopicId` 在 `updatePriority` 之后才写（B2 已改为先写再算）
- source Topic 子树 `children` 拿到旧快照（扫描时对访问节点 drop cache）
- 新创建的 Extract 块拿不到 `created` 时间
- 创建流程没走到 `updatePriority` 的 init/migrate 路径

快速定位：

1. UI 上看**首次 due**是否随创建序号推远（`intervalDays` 应大致同档，不必随序号变大）
2. 检查 `ir.sourceTopicId` 是否指向真实 Topic；临时日志：`blockId` / `sourceTopicId` / `created` / `index` / `stepDays` / `queueDelayDays`
3. 快速创建时 source Topic 子树内 Extract 是否被有界扫描计入

### 6.2 “乱序”是正常的

随机铺开，不要求严格均匀；不挤在 1～2 天窗口即达标。

### 6.3 每日上限 / 时间预算不会自动改排期

`buildIRQueue` + `irQueuePolicy` 负责“今天展示谁”，**不会**自动重排所有卡的 `ir.due`。

需要“当天太多就推迟低优先级”时：用**溢出推后**按钮；全自动 Auto-Postpone 仍为增强方向。

### 6.4 混合会话复习侧

`irMixedQueuePolicy` 混入的是**已到期非新** SRS 复习卡（精确 `due.getTime() <= now`），与 IR 阅读条目交织；不改变 IR 排期写入规则。
