## 2026-07-20：渐进阅读批量移出确认框被吞及面板底部空白

### 问题现象

- 渐进阅读资料库模式下，选中卡片并点击最底部的“批量移出”按钮后，弹出的二次确认框只能看到上半部分，下半部分（含确认/取消按钮）被截断吞掉。
- 设置面板（Drawer）打开时，最下方有约 40% 的空白遮罩和页面没有利用到，且背景模糊和内容在横向上被同步截断。

### 确认原因

1. **容器高度未撑满**：宿主编辑器容器 `.orca-block-editor` 在主面板托管模式（`srs-ir-host-panel-chrome-managed`）下未强制拉伸至 100% 面板高度，导致子组件 `.ir-workspace` 最终缩水在局部高度中。
2. **绝对悬浮导致溢出**：原批量操作栏 `.ir-bulk-bar` 使用 `position: absolute; bottom: 18px;` 悬浮放置。在宿主容器被挤压的情况下，其内部 `ConfirmBox` 二次确认弹窗默认向下展示，触发了宿主 `overflow: hidden` 的剪切边界，从而导致截断。

### 修改路径

- **容器高度自适应与固定底部栏**：
  - `src/styles/ir-workspace.css`：
    - 为 `.orca-block-editor.srs-ir-host-panel-chrome-managed` 增加 `height: 100% !important; overflow: hidden !important;`。
    - 将 `.ir-library-scroll` 的 `padding-bottom` 从原先避开绝对定位悬浮栏的 `88px` 调整为正常的 `16px`。
    - 将 `.ir-bulk-bar` 从 `position: absolute` 改为融入 Flex 流布局的固定底部栏 `position: relative`，增加顶部边框与内阴影，并处理响应式布局下的 `left` / `right` / `bottom` 复位。
- **弹窗定位向上引导**：
  - `src/components/incremental-reading/workspace/IRBulkActionBar.tsx`：
    - 显式为 `ConfirmBox` 传入 `defaultPlacement="top"` 和 `allowBeyondContainer={true}`，引导弹窗安全地在操作栏上方列表空旷区渲染，杜绝下沉截断问题。

## 2026-07-20：Basic 答案区 Tab / Shift+Tab 后 Enter 无法建块

### 问题现象

- SRS 复习面板 → Basic 普通问答卡 → 显示答案 → 编辑答案子块
- Tab / Shift+Tab 后：数据/视觉缩进常已发生，但随后 **Enter 不能创建后续块**；CursorData 可能变 null，DOM selection 离开答案区
- 半成品（已回撤）曾用 window 捕获 + `indentSelection` + indent 前 CursorData 多次恢复：Tab 后可输入，Enter 仍坏

### 候选原因（运行时证据 + 代码）

1. **答案区长期 MutationObserver**（已确认代码路径）：子树 childList 变化 debounce 100ms 后重写 children display、必要时 `collapse.click()`、重写父 `.orca-repr-main` / handle style → 覆盖宿主 Tab/Enter DOM 更新，且 observer 写 DOM 可自触发。
2. **同 panelId+卡根 blockId 双 live Block**：显示答案时题目 `EmbeddedQuestionBlock` 与答案 `EmbeddedAnswerBlock` 同时挂载，可能抢 selection。
3. **过期 CursorData 回写**（半成品策略，勿复用）：制造半有效编辑态，破坏 split/insert 会话。

### 修改路径（第一阶段，自动化已覆盖；Orca 实例待验）

- `EmbeddedReviewBlocks.tsx`：`EmbeddedAnswerBlock` 移除 MutationObserver / timer / collapse.click / style 重写；仅 `Block` + `initiallyCollapsed={false}`。
- `srs-review.css`：直接子选择器只藏答案**根** main/handle/collapse；不藏后代子块 main。
- `BasicCardReviewRenderer.tsx`：`showAnswer && totalChildCount > 0` 时题目改静态 `front`（`.srs-question-static`），只挂答案 live 根；摘录路径不变。
- **不**新增 Tab `preventDefault` / 命令接管 / 旧 CursorData 恢复。
- 回归：`src/components/review-card/reviewBlockExpand.test.ts`
- 只读诊断：`src/test/diagnose-review-tab-focus.js`（Console；失败入 errors）

### 验证边界

- **自动化验证 ≠ Orca 实例验证**。部署后须手测：Tab 缩进后可输入、Enter 建块；Shift+Tab 同理。未在本文宣称「运行时已修复」。

### 经验总结

- 对宿主可编辑子树：**优先 CSS 静态策略** + `initiallyCollapsed`，忌长期 observer 写 style / 伪点击折叠。
- 同一 panel 同一 `blockId` 只挂一份 live `Block`。
- 编辑会话修复忌用 indent 前 CursorData 覆盖选区。

## 2026-07-20：复习面板原笔记折叠后看不到题目

### 问题现象

- 普通问答卡（basic）在原笔记中处于**折叠**时，SRS 复习面板题目区空白
- 展开原笔记节点后，复习界面才能看到题目

### 确认原因

- 题目区 `EmbeddedQuestionBlock` 用宿主 `Block` 渲染卡根
- 未传 `initiallyCollapsed={false}` 时沿用块全局折叠态；`BlockShell` 折叠则 content 不展示
- 答案区 `EmbeddedAnswerBlock` 早已传 `initiallyCollapsed={false}`，题目侧遗漏

### 修改路径

- `src/components/review-card/EmbeddedReviewBlocks.tsx`：题目 `Block` 强制展开
- `BasicCardReviewRenderer` 摘录路径、`ClozeReviewBlockContent`、`SafeBlockPreview`、`ChoiceOptionRenderer`、`CardBlockPreview` 同步
- `src/styles/srs-review.css`：`.srs-question-block .orca-repr-main` 强制 `display:block`
- 回归：`src/components/review-card/reviewBlockExpand.test.ts`

### 经验总结

- 复习/预览等**面板局部**嵌入 `Block` 时默认 `initiallyCollapsed={false}`，不写后端折叠属性
- 与 IR 阅读模式「局部展开」同模式；答案区已做、题目区不得漏

## 2026-07-19：嵌套 Extract 的 sibling queueDelay 为 0（direct-parent Topic 假设）

### 问题现象

- 同一天从同一 Topic 连续创建多个 Extract 后，首次 due 仍挤在相近日期；`queueDelay` 常为 0。
- 真实 Orca 只读证据（repo `sgf9fkhr9z3yz`，`2026-07-19T00:54:15.737Z`，`errors=[]`）：
  - Extract `#1646`：`cardType=extracts`，**parent 为普通正文块 `#237`**，`ir.sourceTopicId=231`，priority=50，`intervalDays≈5.83`，due `2026-07-24`，`lastAction=priority`，`readCount=0`；state/backend 一致。
  - Topic `#231`：priority=50，`intervalDays=8`，`lastAction=init`；state/backend 一致。
- **结论等级**：运行时只读证据已确认「嵌套形状 + sourceTopicId 表达来源」；**不等于**修复后的 Orca 实例验证。

### 确认原因

1. `computeNewExtractQueueDelayDays` 只把**直接 parent** 当 Topic；真实创建形状是 Topic → 正文块 → Extract，parent 非 Topic → 直接返回 0。
2. `createExtract` 在 `updatePriority`（计算 queueDelay）**之后**才写 `ir.sourceTopicId`，且写后无匹配 `invalidateIrBlockCache`，排期读取无法依赖来源属性。
3. 低层 `computeDispersedIntervalDays` 曾把 `queueDelay` 并入 `intervalDays`，即使 delay 非零也会污染后续 ×1.35 复利（与「只影响首次 due」合同不符）。

### 修改路径

- `irSchedulingHelpers.ts`：`resolveExtractSourceTopicId`（优先 `ir.sourceTopicId`，安全回退 direct parent Topic）；有界 BFS sibling 扫描（depth/count/concurrency caps）。
- `extractUtils.initializeExtractScheduleAfterCreate`：invalidate 标签缓存 → 写 sourceTopicId → invalidate 来源属性缓存 → ensure → updatePriority。
- `incrementalReadingDispersal.ts` / `computeDispersedSchedule`：interval 与 due-only offset 分离。
- 顺带 B2：`postpone` / `applyAutoPostpone` / `deferIROverflow` 只移 due；priority 单一路径 + cardType clamp。

### 回归测试

- `src/srs/incremental-reading/irScheduling.queueDelay.test.ts`
- `src/srs/extractUtils.scheduleOrder.test.ts`
- `src/srs/incremental-reading/irScheduling.priority.test.ts`
- `src/srs/incremental-reading/irScheduling.postpone.test.ts`
- `src/srs/incrementalReadingDispersal.test.ts`（queueDelay 不进入 interval）

### 验证边界

- **自动化验证 ≠ Orca 实例验证**。部署插件后仍需在真实库创建嵌套 Extract，核对 `intervalDays` 不含 delay、首次 due 有序推远、推后不改 interval。

## 2026-07-18：网页导入主文污染、代码行号块与 mailto 链接损坏

### 问题现象

- Firecrawl `onlyMainContent: true` 仍带回 MDN/GitHub/Wikipedia/NASA/OpenAI 文档等页的导航、TOC、navbox、页脚相关、Suggested searches 等 chrome。
- 真实 Orca 实例中：文档站导入后根标题下先出现搜索/主导航等外围块，正文标题落在后面；代码块被拆成大量行号块；标识符链接变成 `mailto:` 形态污染。
- 元数据标题带站点后缀（` - Wikipedia`、` | MDN`）时，正文首个 h1 无法去重；盲目去后缀会误伤 `Guide - Part 2`。
- 共享 `sanitizeHtmlForOrca` 去掉全部 `href`；若对同站/引用链接批量附加可见 URL，正文会被上百个 URL 膨胀。
- 合并后的 outline content token 被误报为 `huge_single_chunk`（实际 `batchInsertHTML` 仍按多个 `<p>` 建块）。

### 根因

1. 过度信任 Firecrawl 主文裁剪；本地无第二层提取；rich-structure 回退把导航列表当「有价值结构」而放弃 Readability。
2. 无差别删除 `header`/`footer`/`aside`，误伤 `article` 内标题区与 callout。
3. 语法高亮/行号 gutter 未经规范化即进入 `batchInsertHTML`。
4. 标题去重盲目剥两次短后缀；链接装饰未区分跨域与引用标签。
5. 诊断用合并 content token 长度而非原子块（`p`/`li`/`pre` 等）。

### 解决方案

- `webContentExtract.ts`：chrome 剥离（保留 article 内 header/aside；搜索面板/topic-card 等选择器）+ Readability；结构仍含强 chrome 且 Readability 连贯时**优先 Readability**；对 Readability 结果再做轻量 chrome 清理。
- `webHtmlNormalize.ts` / `webLegacyLayout.ts`：代码折叠；**跨域**描述性链接 `label (https://…)`（上限约 45；同站/引用/`mailto` 仅保留文案）。
- `webTitle.ts`：仅在可信站点后缀匹配后去重/剥后缀。
- `webHtml.ts`：诊断按原子块；`prepareWebArticleHtml` 编排。
- 设置：仅非空显式 `apiKey` 可在 schema 失败时继续；`apiUrl`-only 必须可读设置。

### 回归

- `webHtmlExtract.test.ts` / `webHtmlNormalize.test.ts`（NASA 形态、article header/aside、标题负例、链接上限、huge_single_chunk、代码/mailto）
- `webUrlAndFetch.test.ts`（settings 失败语义、markup-only empty_html）
- EPUB：`sanitizeHtmlForOrca` 不附加 URL

### 验证边界

自动化 + 可选本地 Firecrawl JSON 审计；**不**等于真实 Orca 实例复验。

## 2026-07-18：顺序推进 plan 保存失败导致双 live IR / 会话误关卡

### 问题现象

activate-before-strip 路径下，下一章已初始化但 `saveBookIRPlan` 失败时：计划与真实 IR 不一致，可能两张顺序章同时带 `#card`；重试仅修 plan 不清理旧卡。会话层对所有 `partial` 一律 `leftCard: true`，strip 失败后 UI 仍移除当前卡。

### 根因

- `chapterHasIRScheduling` 以 `ir.due OR #card` 判定「已有 IR」，过宽且不校验 `sourceBookId`
- `retrySequentialActivation` 在 `active==null` 时只写 plan，不 strip 旧 current
- plan 检查点失败用空 `catch`；`performArchive` 不区分 strip 是否成功

### 解决方案

1. **Fully active** = backend `#card` + `ir.due` + 匹配 `ir.sourceBookId`
2. plan-save 失败：可见 partial + 非空错误；检查点失败同时报告两错；`planPersisted` / `currentChapterRemoved` 标志
3. `retrySequentialActivation` 扫描 selected 上本书 IR，保留 plan 顺序最后一章，strip 其余，收敛到单卡
4. 会话 `leftCard` 跟随 `currentChapterRemoved`；`IRSessionShell` 仅在 `leftCard` 时 `removeCurrent()`
5. plan 读 backend-first，写后 `invalidateIrBlockCache` / `invalidateBlockCache`
6. 顺序书注册表发现零 live 卡书籍；章节 id 去重

### 回归

- `src/srs/book-ir/bookIRPlanRepository.test.ts`（stale state vs backend）

## 2026-07-13：子卡展开遇循环引用误报（自身反链）

### 问题现象

复习队列展开子卡时，循环保护报告「子卡展开遇循环引用」，但用户并未构造父子互指。

### 根因

- Orca 卡片块可能存在**自身反链**（`BlockRef` 的 `from === to === parentBlockId`，例如 type=3）。
- `collectChildCards()` 遍历 `parentBlock.backRefs`，把 `backRef.from` 上带 `#Card` 的块都当成子卡。
- 当 `from === parentBlockId` 时，父卡被当成自己的子卡，后续 `cardCollector` 循环保护正确触发，形成**误报**。

### 解决方案

在 `collectChildCards()`（`src/srs/childCardCollector.ts`）中显式跳过 `backRef.from === parentBlockId` 的自身反链；真实子卡反链行为不变。回归：`src/srs/childCardCollector.test.ts`。

### 经验总结

- 反链图中的自环是合法数据，不等于用户配置的子卡层级。
- 循环保护层无法区分「自环误入」与「真循环」；应在收集层过滤自环。

## 2025-12-15：埋藏功能不生效

### 问题现象

用户点击「埋藏」后，`srs.due` 应更新为明天，但重新打开复习面板后卡片仍出现。

### 排查过程

1. 确认 `buryCard` 的 `setProperties` 已写入 `srs.due`
2. 属性类型曾从字符串改为数字标识符，问题仍在
3. 定位到 `storage` 模块级块缓存

### 根因

- `src/srs/storage.ts` 使用模块级 `blockCache` 优化读块
- `buryCard` 位于 `src/srs/cardStatusUtils.ts`，写入后未失效缓存
- 后续 `collectReviewCards` / `ensureCardSrsState` 读到旧缓存

### 解决方案

1. 在 `storage.ts` 导出 `invalidateBlockCache(blockId)`
2. 在 `buryCard` 成功写入后调用失效

### 经验总结

- 跨模块改块数据必须处理缓存一致性
- 写入成功 ≠ 读取路径使用最新值
- 调试日志有助于区分「未写入」与「读到缓存」
