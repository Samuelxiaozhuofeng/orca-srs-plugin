# 仓库贡献与文档维护指南

> **文档同步日期：2026-07-13**（审核修订：根文件名大小写）  
> 对齐仓库根目录 `AGENTS.md`、`CLAUDE.md` 与 `模块文档/README.md` 的文档侧约定（本文件只谈流程，不改代码）。

## 概述

本文件说明开发与维护时的最小协作约定，重点是构建与 **模块文档同步**，避免行为变更未记录或文档滞后。

## 适用范围

- 代码改动：功能、逻辑、交互、配置。
- 文档改动：新增说明、修订流程、修复错误描述。
- 构建输出：影响 `dist/` 或插件加载的变更。

## 模块文档规则（`CLAUDE.md` / `AGENTS.md`）

1. **修改前阅读**：改某功能前先读 `模块文档/` 下对应文档。
2. **修改后更新**：行为或工作流变更后，同步更新对应中文模块文档。
3. **新模块配套文档**：新增模块时在 `模块文档/` 增加文档，并在 `模块文档/README.md` 索引登记。
4. **Orca API**：实现前查 `plugin-docs/`（索引 `plugin-docs/modules.md`），勿臆造未验证 API。
5. **以代码为真相**：文档写现状；计划类文档须标明已落地 / 仍为计划。
6. **禁止编造路径**：`相关文件` 节必须是仓库真实路径（例如 Flash Home 是 `SrsFlashcardHome*.tsx`，**不存在** `SrsCardBrowser.tsx`）。
7. **重复文档**：保留一份权威；另一份顶部注明「请参阅」并精简为摘要/历史。
8. **历史报告/经验**：仅修明显错误；顶部标明「历史文档」。

建议结构（可按模块裁剪）：概述 → 技术实现 → 用户交互 → 配置 → 扩展点 → 测试 → 相关文件。

## 构建与测试（摘自 `AGENTS.md` / 2026-07-19 发布基线）

- `npm install` / `npm ci`：安装锁定依赖（Node **≥ 20.19**）
- `npm run dev`：Vite 开发入口（插件仍需在 Orca 内加载；`index.html` 仅为说明桩）
- `npm test`：Vitest 全量
- `npx vitest run <path>`：单文件
- `npm run build`：类型检查 + **仅仓库内** `dist/` 生产包（**不再**自动复制到本机 Orca 目录）
- `ORCA_PLUGIN_ROOT=… npm run deploy:local`：显式安全本地部署（临时目录校验后替换）
- `npm run release:licenses`：审计生产依赖许可证并生成随包 notices / license texts
- `npm run release:stage` / `release:verify`：固定顶层 `orca-srs/` 发布布局与结构校验
- `npm run release:ready` / `release:zip`：正式门禁；要求 readiness Go、已验收 dist 哈希、干净工作树和 `v<version>` tag
- 基线检查：`npx tsc --noEmit` + 测试套件（仓库未强制统一 lint 命令）
- 当前元数据版本为 `1.0.0`（**版本确认 ≠ Go**）。日常 CI 使用 `release:verify -- --allow-incomplete-readiness` 做结构检查；正式 tag workflow 运行无豁免的 `release:ready` 与 `release:zip`，并上传校验和

## 更新要求

- **功能/修复完成后**：运行 `npm run build`（或至少 `tsc` + 相关测试）确认无报错。
- **模块文档必须同步**（见上）。
- **优先更新中文文档**，索引与 `模块文档/README.md` 一致。
- Commit 风格：历史常用 `feat:` / `fix:` / `refactor:` / `docs:` 等前缀。

## 建议流程

1. 完成功能或修复（必要时先读 `plugin-docs/` 与模块文档）。
2. 更新或新增 `模块文档/` 对应文档；文首/文末可注同步日期。
3. 更新 `模块文档/README.md` 索引条目（关联文件路径写正确）。
4. 运行 `npm test` / 相关 vitest 与 `npm run build`。
