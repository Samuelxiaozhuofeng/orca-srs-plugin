# 闪卡优化方案 2

## 1. 文档用途

本文档是 `闪卡优化方案.md` 完成后的第二阶段工作方案，覆盖本轮代码审查发现的剩余缺口，以及用户确认要新增的三项能力：

1. 断点恢复
2. 自定义学习
3. 每日计划与 7/30 天负载预测

旧的 `闪卡优化方案.md` 保留为第一阶段历史记录，不回改其已完成任务。第二阶段的实施范围、产品规则和验收标准以本文档为准。

## 2. 进度跟踪规则

本方案的唯一进度跟踪文档是：

- `闪卡优化2_PROGRESS.md`

所有执行本方案的 AI Agent 必须遵守以下规则：

1. 开始任务前，先在 `闪卡优化2_PROGRESS.md` 把对应任务状态改为 `in_progress`，填写执行者、开始时间和计划改动范围。
2. 实施期间发现阻塞、范围变化、产品歧义或额外风险时，必须立即写入对应任务的“阻塞/备注”，不得只留在对话中。
3. 只有实现、聚焦测试、`npx tsc --noEmit`、`npm test` 和文档同步全部完成后，才能把状态改为 `done`。
4. 每次任务结束都必须记录实际改动文件、验证命令及真实结果；不得只写“测试通过”。
5. 未真实验证时不得标记完成。Orca UI 手工验证未执行时必须明确保留为待验证项。
6. 方案任务状态只在 Progress 文档更新；本文档不使用任务勾选框，避免出现两个状态来源。
7. Progress 文档中的“待确认设计决策”必须在对应任务开始前解决；执行 Agent 不得把推荐项擅自视为用户已确认。

## 3. 已确认产品规则

以下规则已经由用户确认，实现过程中不得自行改为其他语义。

### 3.1 每日额度是全局总额度

- `review.newCardsPerDay` 和 `review.reviewCardsPerDay` 是所有牌组合计的每日额度。
- 从某个 Deck 启动复习，只改变候选卡范围，不获得一份新的 Deck 独立额度。
- 今日先在 Deck A 学了 10 张新卡，再进入 Deck B 或“全部复习”，这 10 张必须继续占用同一全局新卡额度。
- 多个普通复习会话同时存在时，也不得各自获得完整额度。
- 重复复习、只读预览和不更新 SRS 的专项训练不消耗正式每日额度。

### 3.2 List 正式条目消耗额度

- List 的 Good/Easy 解锁下一条后，该下一条属于正式复习卡，必须按其独立 `cardKey` 消耗新卡或旧卡额度。
- 额度不足时，不得把下一正式条目加入当前队列，也不得提前把它的 due 改成“现在”后假装未加入。
- List 的辅助预览不更新 SRS、不写正式复习日志，也不消耗每日额度。

### 3.3 Again/Hard 保留当前会话短期重学

- Again/Hard 后，如果 FSRS 给出的 due 在短期重学窗口内，卡片应在当前会话到期后再次出现。
- 同一卡片重新入队不重复消耗每日额度。
- 重新入队时只检查“当前索引之后是否已经有待复习副本”，不能因历史位置仍保留该卡就永久拒绝重入。

## 4. 设计原则

### 4.1 数据正确性优先

- 防止重复评分、额度突破和错误卡片范围，比动画或界面细节优先。
- 不用静默降级掩盖日志、存储或 Orca 后端错误。
- 对“块不存在”和“暂时无法确认”使用不同状态。

### 4.2 会话必须有稳定身份

普通复习、Deck 复习、断点恢复和自定义学习都必须基于显式的会话描述对象，至少包含：

- `sessionId`
- 会话类型
- 会话 scope/filter
- 是否更新 SRS
- 是否消耗每日额度
- 创建时间

不得继续用模块级临时变量表达“下一次要启动的会话”。

### 4.3 纯逻辑与 React 状态分开

- 额度接纳、动态追加、List 解锁、短期重入、恢复校验尽量使用可单测的纯函数。
- React state updater 必须保持纯净，不在 updater 内发送通知、修改外部预算对象或写存储。
- 评分、推迟、暂停等持久化动作使用同步 ref 锁，不能只依赖异步 React state。

### 4.4 预测不冒充事实

- 当前卡片的 due 分布属于可计算事实。
- 未来用户会给什么评分、未来 FSRS 会产生什么间隔属于不确定结果。
- 负载预测必须标明“已排期负载”和“情景估算”的区别。
- 调整新卡量或目标保留率的建议必须展示依据，不自动修改设置。

## 5. 阶段 A：修复现有实现缺口

## F2-01 建立显式会话描述与启动交接

**优先级：P0**

### 问题

`startReviewSession()` 把 Deck 范围写入模块级 `reviewDeckFilter`，Renderer 在异步收集卡片后才读取。快速连续启动或多面板并发时，会话可能读取到另一启动请求的范围。

### 实施要求

- 定义版本化的 `ReviewSessionDescriptor`，由启动入口创建并与目标面板/会话块稳定关联。
- Renderer 按当前会话描述加载，不能从全局“最后一次 filter”推断范围。
- normal/all、normal/deck、fixed/repeat、custom 必须有明确且互斥的类型。
- 会话描述读取失败时显示错误，不回退成 all scope。
- 明确同一会话块重复打开时是复用、恢复还是创建新会话。
- 为后续断点恢复提供稳定 `sessionId`，但本任务不实现恢复 UI。

### 验收标准

- 快速启动 Deck A 后再启动 Deck B，两次加载各自使用正确范围。
- Renderer 收集卡片期间全局状态变化不影响当前会话。
- 两个面板不会因为共享临时变量而串 scope。
- fixed/repeat 会话不会误读 normal/deck 描述。
- 有纯逻辑或组件边界回归测试。

## F2-02 修正全局每日额度与并发占用

**优先级：P0**

### 问题

当前 Deck 会话只统计同名 `deckName` 的日志，相当于每个 Deck 获得独立额度；两个同时启动的普通会话也可能基于相同日志各自冻结完整剩余额度。

### 实施要求

- normal/all 和 normal/deck 都从全库今日正式日志计算同一份 global used。
- Deck 只过滤候选卡，不过滤额度使用日志。
- 引入明确的会话占用策略，防止多个活动会话重复预占同一额度。
- 推荐第一版只允许一个“更新 SRS 且消耗额度”的活动会话；启动第二个时提示继续现有会话或结束/放弃现有会话。
- 如果最终选择支持多个活动会话，必须实现可持久化、可释放、可恢复的额度 reservation，不能只靠组件内存 Set。
- 日期跨过本地午夜时重新建立当天额度边界，不能继续沿用前一天冻结值。
- 日志读取或占用状态读取失败时阻止启动并显示错误，不用 used=0 兜底。

### 验收标准

- Deck A 已用额度会减少 Deck B 和 all 会话的剩余额度。
- 第二个普通会话不能获得第一会话已经占用的额度。
- 会话完成、放弃、崩溃恢复后的 reservation 行为有确定规则。
- 跨午夜后使用新一天额度，前一天占用不泄漏。
- fixed/repeat 非排期训练不消耗 global quota。

## F2-03 List 正式条目接入会话预算

**优先级：P0**

### 问题

List Good/Easy 解锁的下一正式条目直接追加到队列，没有经过 `SessionRootCardBudget`，可以突破每日额度。

### 实施要求

- 把“计算下一 List 条目”“尝试占用额度”“写 due”“追加队列”组织成明确事务顺序。
- 先确认 scope 和额度，再写 due 和追加正式条目。
- 若额度不足，保持该条目原有未来 due，并给出简短说明。
- 正式 List 条目按独立 `cardKey` 计入 new/review 对应额度。
- 辅助预览继续不消耗额度。
- 写 due 成功但队列追加失败时必须报告部分失败，并提供可重试或可恢复状态。

### 验收标准

- 新卡剩余额度为 0 时，Good/Easy 不会解锁新的正式 List 新条目。
- 旧卡剩余额度为 0 时，同样不能解锁额外正式旧条目。
- 有剩余额度时只消耗一次，同一身份重试不重复消耗。
- 辅助预览数量不影响 global quota。
- List 多条连续 Good/Easy 不会突破当日总额度。

## F2-04 修复 Again/Hard 短期重新入队

**优先级：P0**

### 问题

当前到期检查使用整个队列去重。已经评分的历史副本仍在 `currentIndex` 之前，因此同一 `cardKey` 永远被判定为已存在，短期重学无法发生。

### 实施要求

- 重入去重只针对当前索引之后尚未处理的队列部分。
- pending 记录按 `cardKey` 幂等更新到最新 due 和最新卡片状态。
- 同一卡在等待期间再次评分时，旧 timer/token 必须失效。
- 到期后重新加入队尾或确定的重学位置，规则写入模块文档。
- 重入不再次消耗 global quota。
- fixed/repeat 是否执行 FSRS 短期重入取决于其是否更新 SRS；纯训练模式不得伪造正式重学。
- 组件卸载、完成和断点保存时正确处理 pending timer。

### 验收标准

- Again/Hard 后卡片按 due 在当前会话再次出现。
- 到期前不会提前出现。
- 已在待复习尾部的同一卡不会重复追加。
- 多次更新 due 后只按最新 due 触发。
- 关闭后没有悬挂 timer 或卸载后 setState。

## F2-05 为评分和状态动作增加同步门闩

**优先级：P0**

### 问题

`setIsGrading(true)` 不能同步阻止同一渲染周期内的第二次点击或快捷键，可能重复推进 FSRS、重复写属性或产生冲突日志。

### 实施要求

- 会话层为 grade/postpone/suspend 等持久化动作增加同步 ref lock 或 token gate。
- gate 在异步调用前同步获取，成功或失败后按确定规则释放。
- 卡片切换、只读回看和组件卸载必须使旧 token 失效。
- Choice 的提交 gate 与正式评分 gate 分工清晰，不能互相替代。
- 重复触发应被忽略或提示，但不能进入第二次持久化。

### 验收标准

- 同 tick 双击评分只调用一次 SRS 更新和一次日志写入。
- 按键自动重复不会重复评分。
- 评分中点击推迟/暂停不会并行写同一卡。
- 第一次失败后用户可以明确重试。
- 切卡后的旧 Promise 完成不会推进新卡。

## F2-06 区分块 missing 与 unknown

**优先级：P1**

### 问题

当前卡块预加载失败时，后端异常和确认不存在都返回 `false`，随后卡片会从队列删除。

### 实施要求

- 复用或抽取 `exists | missing | unknown` 三态结果。
- 只有后端明确返回不存在时才从队列剔除。
- 后端抛错、超时或返回不可判定数据时保留卡片，展示可重试错误。
- List 同时校验父块和条目块，任一 unknown 都不得当 missing。
- 预缓存下一张失败只记录诊断，不影响当前会话。

### 验收标准

- `get-block` 抛错不会自动跳卡。
- 明确不存在的块会安全剔除。
- unknown 状态可重试，错误对用户可见。
- List 父/子读取失败不会误删有效条目。

## F2-07 清理动态追加的 React 状态副作用

**优先级：P1**

### 问题

自动/手动检查在 `setQueue` updater 内修改外部预算、设置其他 state 和通知；手动检查还在 updater 执行前读取局部 `foundNewCards`，可能出现通知与实际队列不一致。

### 实施要求

- 将候选选择和预算变更计算为纯结果：`{ nextQueue, accepted, nextBudget }`。
- 在一次明确提交中更新 ref/state，通知放在提交结果之后。
- React updater 内不发送通知、不写存储、不修改外部 Set。
- 自动检查和手动检查共用同一纯逻辑。
- 检查期间避免并发扫描；后一次请求可复用 in-flight 或明确排队。

### 验收标准

- 找到卡片时通知数量与实际追加数量一致。
- 未找到卡片不会错误覆盖前一次成功提示。
- React Strict Mode 重复调用 updater 不会重复消耗预算或重复通知。
- 自动/手动并发检查不会重复追加。

## F2-08 完整校验 FSRS 设置

**优先级：P1**

### 问题

FSRS 权重只检查 `NaN`，没有拒绝 `Infinity`；目标保留率和最大间隔也没有运行时范围校验。非法设置可能在评分时才报错或生成异常排期。

### 实施要求

- 权重只接受规定数量的有限数字。
- 目标保留率按 UI 说明限制到有效区间，建议 `0.7..0.99`。
- 最大间隔只接受有限正整数，并设置合理安全上限。
- 无效配置必须用户可见，并回退明确默认值；不能只在 console 静默处理。
- 提供恢复默认设置的明确路径。
- 预览间隔和正式评分必须使用同一份已校验参数。

### 验收标准

- `NaN`、`Infinity`、空字符串、错误数量权重均被拒绝。
- 非法 retention/maximum interval 不进入 `ts-fsrs`。
- 回退值和原因对用户可见。
- 预览与实际评分参数一致。

## 6. 阶段 B：断点恢复

## F2-09 设计并持久化完整会话快照

**优先级：P1**

### 目标

会话异常关闭、Orca 重启或面板关闭后，可以恢复完整学习位置，而不是只恢复统计数字。

### 快照最小内容

- schema `version`
- `sessionId` 和 `ReviewSessionDescriptor`
- 创建时间、最后更新时间、本地日期边界
- 队列中每张卡的稳定 `cardKey` 与恢复所需结构化身份
- `currentIndex`
- 已完成动作历史和只读 outcomes
- session progress 统计
- global quota reservation/accepted keys
- List 正式解锁状态
- pending Again/Hard 卡片及其 due
- repeat/custom 模式所需状态

### 实施要求

- 使用能够跨 Orca 重启保留的数据存储；不得依赖只在当前页面生命周期存在的内存状态。
- 实现版本化序列化、严格校验和未来迁移入口。
- 每次已确认的评分/推迟/暂停/正式 List 解锁后保存快照。
- 写入失败必须提示“评分已保存但会话断点保存失败”等准确部分成功状态。
- 保存卡片身份，不把显示文本或数组索引作为身份。
- 当前正在看的卡片耗时在恢复后重新计时，不把离线时间计入复习时长。
- 不保存无法安全序列化的 timer；保存绝对 due，恢复时重新调度。

### 验收标准

- 刷新或重启后快照仍可读取。
- 损坏、未知版本或字段缺失时明确报错，不静默从头开始。
- 日志已写但快照失败时不会回滚已成功评分。
- snapshot 写入有节流/串行策略，不发生旧快照覆盖新快照。

## F2-10 实现恢复、放弃与冲突处理 UI

**优先级：P1**

### 实施要求

- 启动时发现未完成会话，展示“继续上次会话 / 放弃并新建”，不得自动猜测。
- 恢复前重新按结构化身份收集卡片并校验队列。
- 已删除卡片使用 missing 规则跳过并汇总说明；unknown 阻止无提示继续。
- 恢复后已评分卡保持只读，不能再次推进 FSRS。
- 恢复 pending Again/Hard 时按当前时间决定立即入队或重新调度。
- 完成和明确放弃后删除快照并释放 reservation。
- 正常关闭是否保留断点必须在 UI 文案中清晰：建议“关闭并保留进度”和“放弃本次会话”分开。

### 验收标准

- 在第 N 张关闭后恢复，队列、索引、历史、统计和额度一致。
- 恢复后不会重复评分前 N-1 张。
- 恢复时删除一张未完成卡，不会跳到错误身份。
- 完成后不再提示恢复旧会话。
- 放弃后 reservation 被释放，但已完成正式评分仍保留在今日 used 中。

## 7. 阶段 C：自定义学习

## F2-11 自定义学习筛选器与预览

**优先级：P2**

### 第一版筛选能力

- 一个或多个 Deck
- 包含/排除标签
- 卡片类型：Basic、Cloze、Direction、Choice、List、Excerpt
- 状态：新卡、学习中、到期、逾期、未来、困难卡、暂停卡
- due 日期范围
- 最大卡片数
- 排序：到期优先、最久逾期、难度、稳定身份顺序

### 实施要求

- 所有筛选条件生成版本化 `CustomStudyDefinition`。
- 筛选预览显示预计卡片数，并说明其中多少会消耗新卡/旧卡额度。
- 默认不保存预设；保存常用预设作为后续可选增强，不阻塞第一版。
- 多条件组合使用结构化 predicate，不拼接不透明查询字符串。
- List、Cloze、Direction 按独立卡片身份统计。
- 暂停卡默认排除，只有用户明确选择时才展示；是否允许直接复习暂停卡必须在进入会话前说明。

### 验收标准

- Deck/tag/type/state/due 组合筛选结果稳定可重复。
- include/exclude 冲突有明确校验。
- 预览数量与实际启动队列一致，若额度造成截断则明确显示差异。
- 空结果给出原因，不创建空会话快照。

## F2-12 自定义学习会话模式

**优先级：P2**

### 模式定义

第一版提供两种明确模式，不能让用户误解训练是否改变排期。

1. **排期学习**：正式评分，更新 SRS、写正式日志、消耗 global quota。
2. **额外练习**：不更新 SRS，不写正式复习日志，不消耗 global quota；可以记录独立练习统计，但不得混入正式 FSRS 数据。

### 实施要求

- 启动前明确展示模式影响。
- 排期学习复用 normal session 的 quota、action gate、日志和断点恢复规则。
- 额外练习复用 fixed/practice 语义，但使用自己的会话描述，不能依赖全局 repeat session。
- 自定义学习支持断点恢复；恢复时保持原筛选定义和模式。
- 第一版不实现自动保存预设和复杂随机算法，避免扩大范围。

### 验收标准

- 两种模式对 SRS、日志和额度的影响符合定义。
- 额外练习不会改变正式 due。
- 排期学习不能绕过 global quota。
- 恢复时不会把额外练习变成正式排期。

## 8. 阶段 D：每日计划与负载预测

## F2-13 建立 7/30 天负载预测模型

**优先级：P2**

### 输出指标

按天输出：

- 已排期旧卡数量
- 当前逾期积压
- 可引入新卡数量/新卡库存
- 预计复习分钟数
- 按 Deck 的负载分布
- 数据置信度与缺失说明

### 计算规则

- 7 天和 30 天都基于当前卡片 due 生成“已排期负载”。
- 使用稳定 `cardKey` 去重；Cloze、Direction、List 条目独立计数。
- suspended 卡排除。
- 子卡展开不得把同一卡身份重复计入预测。
- List 必须遵守逐条解锁语义；未解锁条目不能全部当作明天正式到期。
- 已逾期卡单独显示，并在计划中形成 backlog，不伪装成未来 due。
- 预计分钟数优先使用用户近 30 天有效单卡时长的稳健统计；数据不足时使用明确标注的默认值。
- 当前 due 分布是事实；未来评分造成的再排期不纳入“精确预测”，只能进入单独的情景估算。

### 验收标准

- 同一数据在 7/30 天聚合中结果确定。
- 日期使用本地日界，覆盖 DST 和月/年边界测试。
- List、变体卡、暂停卡、逾期卡计数符合规则。
- 无历史时长时显示低置信度默认估算。
- 数据读取失败时显示错误，不返回全 0 假装无负载。

## F2-14 每日计划、建议与可视化

**优先级：P2**

### UI 内容

- 今日 global quota：已用/剩余新卡、已用/剩余旧卡
- 未来 7 天日负载图
- 未来 30 天聚合趋势
- 预计每日分钟数
- Deck 负载分布
- 建议的新卡上限
- 必要时给出目标保留率调整建议

### 建议规则

- 新增“每日可用学习时间”设置，作为建议计算的主要约束。
- 推荐新卡量必须考虑逾期积压、未来 7 天峰值和历史单卡有效时长。
- 当旧卡积压或预计分钟数持续超过预算时，优先建议降低新卡量，而不是牺牲到期旧卡。
- 目标保留率建议至少需要足够历史数据；建议门槛暂定为最近 30 天且至少 200 条正式评分日志，最终阈值须在实现任务中用测试固定。
- retention 建议只允许在有效范围内，并展示“提高 retention 会增加复习量，降低 retention 会减少复习量”的方向性影响。
- 不把静态 due 预测包装成 retention 调整后的精确卡片数。
- 建议必须包含依据，例如：“未来 7 天预计日均 28 分钟，高于你的 20 分钟预算”。
- 用户点击应用前展示旧值、新值和影响；默认只建议，不自动修改。

### 验收标准

- 7/30 天图表和文字摘要使用同一预测数据源。
- 改变每日时间预算后建议可重复计算。
- 数据不足时不推荐 retention，只提示继续积累数据。
- 应用建议需要明确确认，并通过设置校验。
- 首页额度用尽时显示“今日计划已完成”，不显示“没有卡，请先创建”。
- 移动/窄面板下图表、标签和数值不重叠。
